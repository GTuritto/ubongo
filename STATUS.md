# Ubongo — Implementation Status

Last updated: 2026-05-31

## Overall

Phase 18 (GP Loop, autonomous) is complete and merged into `main` via PR #10 (merge commit `40fe1fa`) — **Tier 5 (Self-Improvement), third phase**. The optimize and evaluate halves connect into a continuous background loop. `evolution/loop.py` splits three concerns: `run_one_cycle(*, budget)` is pure and synchronous (picks the stalest target via `evolution/selection.py::next_target`, generates a generation seeded from the previous one's champion survivor — cross-generation lineage with `parent_id` set — or re-evaluates an interrupted generation, evaluates under a shared `CallBudget`, records an `evolution_runs` row, emits `evolution_generation`); `_should_cycle(...)` is the pure gate (status / rolling-hour budget / cron interval); `EvolutionLoop` is a daemon thread running an asyncio loop that drives `run_one_cycle` when the gate allows (sleep + tick injectable for tests). The REPL starts the loop when `evolution.enabled` and stops it on every exit path (try/finally around the input loop); it comes up **paused** (persisted status), so nothing spends until `/evolution resume`. The Phase 17 per-run cap becomes a real rolling-hour window: `evolution_runs` logs each cycle's `calls_spent` + `ended_at`, and `store.calls_in_last_hour` seeds each cycle's budget from `max_calls_per_hour - that`. `evolution.cron` (`null` = continuous, int = min seconds between cycles) paces it; `evolution.survivors` (default 3) sets top-K. Crash recovery is derive-from-DB — an unevaluated latest generation is re-evaluated, not regenerated. `generator.generate` gained optional `budget` (gates LLM strategies; perturb stays free) + `parent_text`/`parent_id` (mutate from a survivor); `lineage.record_variants` honors per-variant `parent_id`; both keep Phase 16 behavior when omitted. New `/evolution status|pause|resume|off` REPL command; `persist_cohort_evaluations` (the rank+write step) is shared by the loop and `/evaluate`. Two new tables (`evolution_runs`, `evolution_state`) via `CREATE TABLE IF NOT EXISTS` — no ALTER migration. 623 / 623 pytest green (584 + 39 new across `test_evolution_selection`, `test_evolution_loop_cycle`, `test_evolution_throttle`, `test_evolution_control`, `test_evolution_recovery`, `test_repl_evolution`, plus generator/lineage extensions); Phase 18 smoke section (18.1–18.12) added. Promotions / acting on winners are Phase 19.

Phase 17 (Sandboxed Evaluation + Fitness) is complete and merged into `main` via PR #9 (merge commit `e0eba6a`) — **Tier 5 (Self-Improvement), second phase**. The optimize→evaluate half of the GP loop closes: `/evaluate <target>` scores a target's latest generation of variants against a held-out conversation set and prints a fitness-ranked leaderboard. New `evolution/sandbox.py` is an offline harness with **no** side effects (no workflow_runs / agent_runs / governance / vault / queue / memory writes — only `evolution_evaluations` rows): for each sample it generates a response (system prompt = `UBONGO.md` + variant text, no skill/memory/agent-role layers, to isolate the variant) then runs one judge call returning `{quality, hallucination, would_user_correct}` — a dedicated 3-signal judge, separate from the governance `EvaluatorAgent`, which stays untouched. `evolution/fitness.py` min-max normalizes cost/latency across the cohort, applies the weighted sum from `evolution.fitness_weights`, and ranks deterministically (fitness desc, then `lineage_id` asc — the tiebreak). A `CallBudget` seeded from `evolution.max_calls_per_hour` caps LLM calls per run and returns partial results when exhausted (variants are all-or-nothing so the cohort stays comparable); new `evolution.samples_per_eval` (default 5) caps samples per variant. The held-out set `tests/manual/fixtures/sample_conversations.json` ships 33 anonymized conversations (8 architect, 8 operator, 7 casual, 6 general, 4 hallucination traps). `store.py` gains `append_evaluation` / `evaluations_for_target` (joins lineage, ranked best-first) / `latest_evaluation_for_lineage`. The `/evaluate <target>` REPL command (a direct tool like `/optimize`) renders the leaderboard; `/evaluate` with no arg lists targets that have variants. No schema migration (`evolution_evaluations` and `fitness_weights` already shipped). 583 / 583 pytest green (549 + 34 new across `test_evolution_fitness`, `test_evolution_sandbox`, `test_evolution_evaluations`, `test_repl_evaluate`); Phase 17 smoke section (17.1–17.12) added. Cross-run hourly rate limiting and the autonomous loop are Phase 18; promotions are Phase 19.

Phase 16 (Variant Generation) is complete and merged into `main` via PR #8 (merge commit `038f10d`) — **start of Tier 5 (Self-Improvement)**. The GP layer the v0.1 vision is named for begins here, with on-demand variant generation only (evaluation is Phase 17, the autonomous loop is Phase 18, promotions are Phase 19). A new `src/ubongo/evolution/` package ships three modules: `targets.py` (the evolvable-target registry — Phase 16 scopes to the three persona prompts, addressed `persona:architect|operator|casual`; `resolve_base` returns the promoted active variant when one exists, else the persona file body; `peer_of` gives each a recombine partner), `generator.py` (five mutation strategies — paraphrase, prune, expand, recombine, perturb_temperature — allocated round-robin over `population_size` so a run is never all-paraphrase; `perturb_temperature` is a pure-metadata variant with no LLM call; a strategy that fails or can't run is skipped and the round-robin backfills, so the count holds and the run never crashes), and `lineage.py` (`record_variants` computes the next generation as `max(existing)+1` and resolves `parent_id` from `active_evolutions` — always NULL in Phase 16 since no promotions exist yet). The `evolution_lineage` table and the `settings.evolution` block already shipped, so there is **no migration**. `store.py` gains `append_lineage_variant` / `lineage_for_target` / `max_lineage_generation` / `active_lineage_id`; `config.load_evolution()` is a thin accessor over the settings block. The `/optimize <target>` REPL command generates, persists, and previews variants (a direct tool like `/exec` — no `master.handle`, no governance, no enqueue); `/optimize` with no arg lists the evolvable targets. 549 / 549 pytest green (515 + 34 new across `test_evolution_targets`, `test_evolution_generator`, `test_evolution_lineage`, `test_repl_optimize`); Phase 16 smoke section (16.1–16.10) added.

Phase 15 (Approval Gates + Sandboxing) is complete and merged into `main` via PR #6 (merge commit `1137d3e`) — **end of Tier 4 (Governance)**. Phase 14's `require_approval` decision becomes an interactive gate. `governance/approval.py` turns a `require_approval` `Decision` into an `ApprovalRequest` (`build_request` / `explain` — a one-line summary + a "why" paragraph); `master.handle` attaches it to `Response.approval` and gains an `approved` parameter (an approved re-issue overrides `require_approval` → `auto` with reason `approved_by_user`). The REPL `_prompt_approval` reads `y/n/why` — `why` prints the explanation and re-prompts, `n` records the decline and aborts, `y` records approval and re-issues the turn with `approved=True` so the real answer is delivered. `store.update_governance_decision(decision_id, approval_response)` patches the choice onto the `governance_decisions` row written earlier in the turn. One-shot is non-interactive — a `require_approval` turn prints the gated message and exits `rc=1`. The Execution Agent's sandbox is hardened (Phase 15c): the parent resolves each allowlisted command to an absolute path at import (`_PROGRAM_PATHS`) and the child subprocess runs with `PATH=""` so it cannot spawn further programs by bare name; `_check_paths` gains a filesystem allowlist (any absolute-path argument must resolve inside the repo tree). New `docs/SECURITY.md` documents the full contract and its known v0.1 limits (no OS-level isolation; network governed by the allowlist, not blocked). 515 / 515 pytest green; Phase 15 smoke section (15.1–15.12) passes.

Phase 14 (Risk + Confidence Scoring) is complete and merged into `main` via PR #5 (merge commit `a795f1e`) — **start of Tier 4 (Governance)**. Its full testing plan and smoke playbook (scenarios 14.1–14.14, 491 pytest green) pass. The decision matrix actually decides. Three score modules feed `governance/decision.py::decide()`: `governance/risk.py` (`score_risk` — the higher of the classifier's `risk` and a destructive-keyword scan of the message), `governance/confidence.py` (`score_confidence` — the Evaluator's score, classifier confidence as fallback), `governance/reversibility.py` (`score_reversibility` — `irreversible` when the workflow runs the `execution` agent or an irreversible skill). `decide()` loads `config/governance.yaml` and applies a 5-rule matrix in priority order: destructive → `require_approval`; high + irreversible → `require_approval`; evaluator confidence below the reject floor → `reject`; `command` turn with low classifier confidence → `ask_clarification`; else `auto`. `master.handle` acts on all four actions — a `_GATED_MESSAGES` map replaces the response with a canned message for `reject` / `ask_clarification` / `require_approval` (the interactive y/n approval flow is Phase 15). `governance_decisions` now persists `reversibility` (no longer NULL) plus the scored `risk`/`confidence`; `/decisions` and `/trace` render the `rev` column. New `/policy` REPL command prints the live matrix. The Phase-10 `CRITIC_LOW/HIGH` constants moved into `governance.yaml::thresholds.critic_band`; the `governance:` block left `settings.yaml` for `governance.yaml` (single config home), and `router._confidence_threshold()` (the persona-switch hysteresis floor) reads `thresholds.auto_route_min_confidence` from there. 491 / 491 pytest green; full Phase 0–14 smoke playbook scenarios pass.

Phase 13 (Repair Agent Activated) is complete and merged into `main` via PR #4 (merge commit `1ecbf71`) — **end of Tier 3 (Self-Healing)**. Its full testing plan and smoke playbook (scenarios 13.1–13.17, 454 pytest green) pass. The Repair Agent now classifies agent failures into one of seven `FailureKind` categories (`timeout | model_error | parse_error | content_rejection | precondition_missing | infinite_loop | unrecoverable`) and returns an ordered `RecoveryPlan` via `plan_recovery(failed_agent, original, attempts_so_far)`. The runner's sequential mode walks the full strategy ladder (variant prompt → different model → smaller model + shorter prompt → peer replacement → abort) capped at `agents.repair.max_attempts=3`. All five fan-out modes (parallel, competitive, collaborative, debate, speculative) use peer replacement only via `_maybe_replace_failed` — cancel-and-retry in `asyncio.gather` is still ambiguous, but a single peer substitution is a clean one-hop replacement; speculative scopes replacement to the cheap leader (strong is the natural fallback). **`PRECONDITION_MISSING`** (Option A) is the new "input contract not met" bucket: `critic_no_candidate`, `memory_missing_input`, `execution_no_command` skip variant-prompt retries and lead with peer replacement, which directly fixes smoke 12.4's `critic_no_candidate` failure (critic in collaborative mode gets architect substituted, restoring the merged-doc's critic-slot section under `## persona composer`). `config/settings.yaml::agents.repair.peer_replacements` ships defaults: `coding/research/critic → architect`, `architect ↔ operator`, `casual → operator`; `evaluator/memory/execution → null`. New `WriteBuffer` + `workflow_buffer()` context manager (`memory/write_buffer.py`) formalizes commit-on-success: `master.handle` stages the assistant-message commit and either `buf.commit()` on `result.ok` or `buf.drop()` on failure; Phase 19/20 agents inherit the seam for their own staged writes. New `repair_runs` audit table (FK → workflow_runs) persists one row per strategy attempted with `failure_kind`, `original_error`, `strategy_attempted`, `peer_agent`, `override_model`, `attempt_index`, `outcome ∈ {recovered, failed, aborted}`, timings. `/trace` renders an indented `repair: kind=… strategy=… outcome=… peer=… model=…` line under the failing `agent_runs` row. `workflow_runs.outcome='repaired'` lights up when any repair recovered AND `result.ok=True`. When the ladder exhausts, `master.handle` returns `Response` with `requires_user_decision=True` and a `repair_summary` dict; REPL prompts `Retry the same message? (y/n)` (one-shot retry; chained retries are Phase 14 governance scope); one-shot prints the apology and exits `rc=1`. Per-agent prompt-hint plumbing (one-line addition per persona/coding/research/evaluator/critic): reads `input.metadata['repair_prompt_hint']` and appends under `## Repair guidance`; reads `input.metadata['max_tokens_override']` to cap retries. Phase 11's `plan_retry` shim retained for back-compat. Phases 0–13 merged into `main`. Phase 12 (Execution Modes) landed via merge commit `ac30993` from PR #3 — **end of Tier 2 (Multi-Agent System)**. The WorkflowRunner now supports all six execution modes (sequential, parallel, competitive, collaborative, debate, speculative). Internally async (each mode is a strategy coroutine selected off `workflow.execution_mode`) but its public `execute()` stays sync via `asyncio.run`, so `master.handle` and the REPL stay sync. Sequential agents run serially with Phase 11 Repair retry; parallel/competitive/collaborative use `asyncio.gather` (no Repair in fan-out modes). `EvaluatorAgent` gained `rank()` (competitive winner picker; returns `{winner, winner_index, reason, scores}`) and `agree()` (speculative agreement check; bool/None). `Workflow` dataclass gained optional `rounds` (debate; default 2) and `timeout_s` (speculative; default 10). PersonaAgents read `debate_role` from `input.metadata` (`"challenge"` for round-2+ debaters, `"synthesize"` for the synthesizer). `/mode <workflow>` REPL command pins a workflow for the next turn (one-shot; mirrors `/skill`); `/mode list` prints every declared workflow with its mode + agents. `master.plan` skips the Phase-10 evaluator-append for competitive (the trailing evaluator is part of the mode contract). Five new example workflows in `workflows.yaml`: `research_brief_parallel`, `coding_competitive`, `brief_collaborative`, `debate_then_synthesize`, `speculative_brief`. None of the new modes are auto-routed via `routing.yaml` in Phase 12 — users opt in via `/mode`. Phase 11 (Coding + Execution + Repair Agents) landed via merge commit `2d370f0` from PR #2 (local merge; not pushed to origin). Phase 11 lands the last three workers and lights up the `coding_session` workflow: **CodingAgent** (`composer=True`, `models.coding`, code-first system prompt — write the function, type hints + docstring + usage example, name assumptions); **`src/ubongo/sandbox.py`** owns the entire safety contract for shell execution (18-command read-mostly allowlist, no shell metacharacters anywhere, no path traversal, `shell=False`, restricted PATH, repo-root cwd, 10s default timeout, stdout/stderr capped at 2KB/1KB) — the SKILL.md body for `constrained-bash` is metadata + a prompt template, never the enforcement; **ExecutionAgent** (`composer=False`) bridges target command → sandbox; **RepairAgent** (`composer=False`) lives in the registry but never runs as a workflow step — the runner consults `RepairAgent.plan_retry` on `agent_failed` and re-dispatches the failing agent ONCE with a model fallback (per-agent fallback model resolved from `agents.repair.fallback_models` in `settings.yaml`, defaults to `models.default` except `casual` which stays on cheap). `agent_runs.retried` boolean column added with a migration shim; `/trace` renders `(retried)` on those rows. `override_model` plumbed through every LLM-calling agent via `AgentInput.metadata`. `/exec <cmd>` REPL command bypasses `master.handle` for direct sandbox debug — no `workflow_runs`, no governance, no enqueue, no vault. `coding_session` workflow now `("coding", "architect")` with `evaluate=true` — last-composer-wins (Phase 10 rule) makes the architect's wrap the user-facing response. `execution_session` declared in `workflows.yaml` but NOT auto-routed (Phase 15 approval gate first). Phase 10 (Evaluator + Critic + Persona Agents Formalized) lands three new workers and one rename: `EvaluatorAgent` is LLM-as-judge over the persona's response (`{confidence: float, issues: [str]}`); `CriticAgent` provides a contrarian frame and is invoked by Master in one path when evaluator confidence falls in `[0.2, 0.6)` — a second pass `(critic, persona)` runs under the same `workflow_run_id` and the retry's text becomes the response. Persona Agents are now class-based (`ArchitectPersona` / `OperatorPersona` / `CasualPersona` subclassing `BasePersonaAgent`) and live in the registry under bare names (`architect`, not `persona:architect`); `workflows.yaml` is updated accordingly. `workflows.yaml` gains a per-workflow `evaluate: true|false` flag; when true Master appends `evaluator` to `workflow.agents` at plan time. `governance.decide` rejects when evaluator confidence is below `0.2` (reason `evaluator_confidence_below_floor:X.XX`); everything else still returns `auto` (real matrix lands Phase 14). `governance_decisions.confidence` now stores the evaluator's score when present, classifier's confidence as fallback. The runner gained a `composer` attribute (read via `getattr`): the user-facing `WorkflowResult.text` is the last composer agent's text, not the last validator's. `WorkflowResult.evaluator_confidence: float | None` is the new field on the result dataclass. The REPL `/trace [n]` command (default N=1) prints recent `workflow_runs` joined with their `agent_runs` and `governance_decisions` — classification, workflow.agents, per-agent timings + tokens + confidence, governance action. v0.1 scope is multi-agent + self-improving + CLI; see [UBONGO_BUILD.md](UBONGO_BUILD.md).
