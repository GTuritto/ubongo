# Ubongo ‚Äî Implementation Status

Last updated: 2026-06-12. Current version: **v0.1.5** (the v0.1 build, plus the post-v0.1 web UI (v0.1.1), self-authored-skills (v0.1.2), local profiler + service control (v0.1.3), MCP server channel (v0.1.4), and MCP client / Connector agent (v0.1.5) layers; v0.2 is Telegram).

## Overall

**v0.1 is COMPLETE.** All 22 phases (0‚Äì21) are built, merged to `main`, and certified: the full cumulative smoke test passes end-to-end (723 / 723 pytest at certification; 960 / 960 green on current `main` after the post-v0.1 layers, re-certified end-to-end on 2026-06-11 with the profiler armed), all 26 v0.1 acceptance criteria below are met, and the codebase is ~15,400 LOC under `src/` (under the ~15,000 soft target). Ubongo is a multi-agent orchestration platform with a six-mode workflow runner, a self-healing repair ladder, a governance decision matrix with an interactive approval gate, a closed human-approved genetic-programming self-improvement loop (prompts + routing/tool-chain/retry config), semantic memory (sqlite-vec recall + a vault-link graph), and bidirectional vault sync with a unified audit log. Built across 16 PRs, each phase on its own branch, merged only after review. The next work is **v0.2 (Telegram)** ‚Äî a new version, additive on the existing event/queue seams, out of v0.1 scope.

Phase 14 (Risk + Confidence Scoring) is complete and merged into `main` via PR #5 (merge commit `a795f1e`) ‚Äî **start of Tier 4 (Governance)**. ## Post-v0.1 ‚Äî self-authored skills (2026-06-09)

The self-extension experiment ‚Äî the last open item in
[docs/ubongo-open-items.md](docs/ubongo-open-items.md). The GP loop *tunes* what
exists; this lets Ubongo *author brand-new skills* behind a human approval boundary
([ADR-0013](docs/adr/0013-self-authored-skills-quarantine-and-approval.md)). A new
`src/ubongo/authoring/` package (~1,285 LOC) mirrors `evolution/` one-for-one across
five branch-per-phase steps, each merged to `main` after a live smoke:

- [x] **Phase 1** ‚Äî candidate model + LLM drafting + validation (schema reuse + a
  command-skill risk floor) + quarantine (`config/skills_candidates/`, not scanned by
  `skills.py`) + the `authored_skills` table + manual `/author`. `sandbox.validate_command`
  extracted so a generated command template is vetted against the exact run-time contract.
- [x] **Phase 2** ‚Äî side-effect-free `evaluate_candidate` (prompt judge + command dry-run)
  + a deterministic `score_candidate` scalar, surfaced in `/author` and `/skill-candidates`.
- [x] **Phase 3** ‚Äî the approval gate `/skill-candidates approve|reject|rollback` with
  **versioned backups** (a re-author backs up the prior version; rollback restores it).
- [x] **Phase 4** ‚Äî the autonomous authoring daemon (`AuthoringLoop`): boots paused,
  budget-throttled, infers recurring capability gaps and drafts into quarantine; approval
  stays manual. `/authoring status|pause|resume|off`.
- [x] **Phase 5** ‚Äî docs: this entry, ADR-0013, the SECURITY.md threat model, the
  CONTEXT/STATE/README updates, and the authoring smoke-playbook section.

The boundary holds at four layers: quarantine before discoverability, a code-enforced risk
floor (not author-declared), static command validation reusing the sandbox contract, and a
daemon that only ever drafts. Nothing self-authored is discoverable or runnable until a human
approves it; the sandbox allowlist stays a human-only change (ADR-0005). Verified at each merge:
full pytest green (‚âà874) + the Phases 0‚Äì21 cumulative smoke + the authoring lifecycle live
(manual `/author`, the approval gate, and the autonomous daemon).
