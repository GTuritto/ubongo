# Ubongo — Implementation Status

Last updated: 2026-05-15

## Overall

Phase 13 (Repair Agent Activated) is implemented on `phase-13-repair` and awaiting user merge — **end of Tier 3 (Self-Healing)**. The Repair Agent now classifies agent failures into one of seven `FailureKind` categories (`timeout | model_error | parse_error | content_rejection | precondition_missing | infinite_loop | unrecoverable`) and returns an ordered `RecoveryPlan` via `plan_recovery(failed_agent, original, attempts_so_far)`. The runner's sequential mode walks the full strategy ladder (variant prompt → different model → smaller model + shorter prompt → peer replacement → abort) capped at `agents.repair.max_attempts=3`. Parallel and collaborative modes use peer replacement only via `_maybe_replace_failed` — cancel-and-retry in `asyncio.gather` is still ambiguous. **`PRECONDITION_MISSING`** (Option A) is the new "input contract not met" bucket: `critic_no_candidate`, `memory_missing_input`, `execution_no_command` skip variant-prompt retries and lead with peer replacement, which directly fixes smoke 12.4's `critic_no_candidate` failure (critic in collaborative mode gets architect substituted, restoring the merged-doc's critic-slot section under `## persona composer`). `config/settings.yaml::agents.repair.peer_replacements` ships defaults: `coding/research/critic → architect`, `architect ↔ operator`, `casual → operator`; `evaluator/memory/execution → null`. New `WriteBuffer` + `workflow_buffer()` context manager (`memory/write_buffer.py`) formalizes commit-on-success: `master.handle` stages the assistant-message commit and either `buf.commit()` on `result.ok` or `buf.drop()` on failure; Phase 19/20 agents inherit the seam for their own staged writes. New `repair_runs` audit table (FK → workflow_runs) persists one row per strategy attempted with `failure_kind`, `original_error`, `strategy_attempted`, `peer_agent`, `override_model`, `attempt_index`, `outcome ∈ {recovered, failed, aborted}`, timings. `/trace` renders an indented `repair: kind=… strategy=… outcome=… peer=… model=…` line under the failing `agent_runs` row. `workflow_runs.outcome='repaired'` lights up when any repair recovered AND `result.ok=True`. When the ladder exhausts, `master.handle` returns `Response` with `requires_user_decision=True` and a `repair_summary` dict; REPL prompts `Retry the same message? (y/n)` (one-shot retry; chained retries are Phase 14 governance scope); one-shot prints the apology and exits `rc=1`. Per-agent prompt-hint plumbing (one-line addition per persona/coding/research/evaluator/critic): reads `input.metadata['repair_prompt_hint']` and appends under `## Repair guidance`; reads `input.metadata['max_tokens_override']` to cap retries. Phase 11's `plan_retry` shim retained for back-compat. Phases 0–12 merged into `main`. Phase 12 (Execution Modes) landed via merge commit `ac30993` from PR #3 — **end of Tier 2 (Multi-Agent System)**. The WorkflowRunner now supports all six execution modes (sequential, parallel, competitive, collaborative, debate, speculative). Internally async (each mode is a strategy coroutine selected off `workflow.execution_mode`) but its public `execute()` stays sync via `asyncio.run`, so `master.handle` and the REPL stay sync. Sequential agents run serially with Phase 11 Repair retry; parallel/competitive/collaborative use `asyncio.gather` (no Repair in fan-out modes). `EvaluatorAgent` gained `rank()` (competitive winner picker; returns `{winner, winner_index, reason, scores}`) and `agree()` (speculative agreement check; bool/None). `Workflow` dataclass gained optional `rounds` (debate; default 2) and `timeout_s` (speculative; default 10). PersonaAgents read `debate_role` from `input.metadata` (`"challenge"` for round-2+ debaters, `"synthesize"` for the synthesizer). `/mode <workflow>` REPL command pins a workflow for the next turn (one-shot; mirrors `/skill`); `/mode list` prints every declared workflow with its mode + agents. `master.plan` skips the Phase-10 evaluator-append for competitive (the trailing evaluator is part of the mode contract). Five new example workflows in `workflows.yaml`: `research_brief_parallel`, `coding_competitive`, `brief_collaborative`, `debate_then_synthesize`, `speculative_brief`. None of the new modes are auto-routed via `routing.yaml` in Phase 12 — users opt in via `/mode`. Phase 11 (Coding + Execution + Repair Agents) landed via merge commit `2d370f0` from PR #2 (local merge; not pushed to origin). Phase 11 lands the last three workers and lights up the `coding_session` workflow: **CodingAgent** (`composer=True`, `models.coding`, code-first system prompt — write the function, type hints + docstring + usage example, name assumptions); **`src/ubongo/sandbox.py`** owns the entire safety contract for shell execution (18-command read-mostly allowlist, no shell metacharacters anywhere, no path traversal, `shell=False`, restricted PATH, repo-root cwd, 10s default timeout, stdout/stderr capped at 2KB/1KB) — the SKILL.md body for `constrained-bash` is metadata + a prompt template, never the enforcement; **ExecutionAgent** (`composer=False`) bridges target command → sandbox; **RepairAgent** (`composer=False`) lives in the registry but never runs as a workflow step — the runner consults `RepairAgent.plan_retry` on `agent_failed` and re-dispatches the failing agent ONCE with a model fallback (per-agent fallback model resolved from `agents.repair.fallback_models` in `settings.yaml`, defaults to `models.default` except `casual` which stays on cheap). `agent_runs.retried` boolean column added with a migration shim; `/trace` renders `(retried)` on those rows. `override_model` plumbed through every LLM-calling agent via `AgentInput.metadata`. `/exec <cmd>` REPL command bypasses `master.handle` for direct sandbox debug — no `workflow_runs`, no governance, no enqueue, no vault. `coding_session` workflow now `("coding", "architect")` with `evaluate=true` — last-composer-wins (Phase 10 rule) makes the architect's wrap the user-facing response. `execution_session` declared in `workflows.yaml` but NOT auto-routed (Phase 15 approval gate first). Phase 10 (Evaluator + Critic + Persona Agents Formalized) lands three new workers and one rename: `EvaluatorAgent` is LLM-as-judge over the persona's response (`{confidence: float, issues: [str]}`); `CriticAgent` provides a contrarian frame and is invoked by Master in one path when evaluator confidence falls in `[0.2, 0.6)` — a second pass `(critic, persona)` runs under the same `workflow_run_id` and the retry's text becomes the response. Persona Agents are now class-based (`ArchitectPersona` / `OperatorPersona` / `CasualPersona` subclassing `BasePersonaAgent`) and live in the registry under bare names (`architect`, not `persona:architect`); `workflows.yaml` is updated accordingly. `workflows.yaml` gains a per-workflow `evaluate: true|false` flag; when true Master appends `evaluator` to `workflow.agents` at plan time. `governance.decide` rejects when evaluator confidence is below `0.2` (reason `evaluator_confidence_below_floor:X.XX`); everything else still returns `auto` (real matrix lands Phase 14). `governance_decisions.confidence` now stores the evaluator's score when present, classifier's confidence as fallback. The runner gained a `composer` attribute (read via `getattr`): the user-facing `WorkflowResult.text` is the last composer agent's text, not the last validator's. `WorkflowResult.evaluator_confidence: float | None` is the new field on the result dataclass. The REPL `/trace [n]` command (default N=1) prints recent `workflow_runs` joined with their `agent_runs` and `governance_decisions` — classification, workflow.agents, per-agent timings + tokens + confidence, governance action. v0.1 scope is multi-agent + self-improving + CLI; see [UBONGO_BUILD.md](UBONGO_BUILD.md).

## v0.1 Acceptance Criteria

- [ ] CLI REPL responds; one-shot command runs and exits.
- [ ] Manual `/architect`, `/operator`, `/casual` commands work and feel different.
- [ ] In `/auto` mode, persona is selected automatically and feels mostly right.
- [ ] You can correct auto-selection with a slash command.
- [ ] `UBONGO.md` is loaded for every persona; editing it changes behavior across all personas after `/reload`.
- [ ] Conversation context persists across CLI restarts within a session.
- [ ] New session starts after 30 minutes of inactivity.
- [ ] Compaction kicks in past the configured threshold; older history replaced by a summary in recall; summary persisted and not regenerated.
- [ ] Daily notes write to the Obsidian vault and render correctly.
- [ ] `summarize-conversation` skill works via `/summary`. Skill bodies not loaded until activation.
- [ ] `/reload` picks up edits to `UBONGO.md`, personas, and skill metadata without restart.
- [ ] Every outbound message goes through `notification_queue`.
- [ ] Master Agent classifies, plans, dispatches, governs, composes per turn; `/decisions` and `/trace` populated.
- [ ] All eight worker agents (Research, Coding, Evaluator, Repair, Memory, Critic, Execution, Persona) registered and dispatchable; `/agents` lists them.
- [ ] All six execution modes (sequential, parallel, competitive, collaborative, debate, speculative) selectable via `/mode`.
- [ ] Repair Agent recovers timeouts, parse errors, agent failures; rollbacks leave no partial state.
- [ ] Decision matrix returns auto / ask_clarification / require_approval / reject per `governance.yaml`; `governance_decisions` populated.
- [ ] `require_approval` flow prompts user; y/n/why all work; Execution Agent properly sandboxed.
- [ ] `/optimize <target>` generates variants; `/evaluate` produces a fitness leaderboard.
- [ ] GP loop runs autonomously when enabled; throttled; pauseable.
- [ ] `/improvements` lists pending promotions with diffs; approve/reject/rollback work; live-target swap takes effect.
- [ ] Semantic recall via `sqlite-vec` augments recency in `/recall`; vault-link graph queryable.
- [ ] File watcher ingests vault edits; conflicts gated by approval flow.
- [ ] Full `tests/manual/smoke_test.md` walkthrough passes end-to-end.
- [ ] Total project size stays under ~15,000 lines of Python (excluding tests).
- [ ] Each phase landed via its own branch and was merged to `main` only after user approval.
