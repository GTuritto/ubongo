# Ubongo — Architectural Snapshot

*2026-06-20, written from the code on branch `v0.6/00-streaming-seam` (`main` is at v0.5.7, the trust protocol complete). ~17,393 LOC under `src/ubongo/`, 1,057 tests, 23 accepted ADRs. Where docs and code disagree, the code is reported.*

## 1. Identity and purpose

Ubongo is a single-user, local-first AI mind: a multi-agent orchestration runtime and a self-improving system in one package. A Master Agent runs every user turn through a fixed governed pipeline, dispatching disposable worker agents across six execution modes; a genetic-programming loop continuously evolves the system's own prompts and routing/tool-chain config, a second loop drafts new skills, and a third now runs proactive scheduled jobs — but nothing the system produces about itself, and no new external reach, goes live without explicit human approval. The problem it solves is personal: one user wants an assistant whose behavior is inspectable, governable, and improvable, without ceding any of it to a SaaS or a framework. What began as a CLI is now reachable over six channels (REPL, one-shot, Streamlit web, MCP server, Telegram, and a streaming browser console), each an additive adapter over one turn seam.

## 2. Component map and data flow

Every turn, from any channel, takes the same path. The channel layer is presentation only:

```
repl.py / oneshot.py / web/app.py / mcp/server.py / telegram/bot.py / web/console/app.py
        └── channel.py            bootstrap() + run_turn(): the one turn envelope
              └── master.py       Master.handle(): classify → plan → execute → govern
                                  → compose → commit → enqueue
```

[channel.py](src/ubongo/channel.py) is the no-bypass rule as code (ADR-0002/0003): `bootstrap()` loads config + logging once and resolves the `UBONGO_PROFILE` knob; `run_turn()` optionally wraps the turn in cProfile, calls `master.handle` (resolved at call time so tests can patch it), and flushes the outbound queue. It starts no daemons. Each front keeps only what differs: printing/exit codes (one-shot), rendering + Approve/Deny (web), TypedDict shaping (MCP), long-poll + auth (Telegram), and — newest — a background-thread turn whose events stream to a browser (console).

Inside the pipeline: [classifier.py](src/ubongo/classifier.py) yields a tone/intent classification with a deterministic fallback (ADR-0009). [router.py](src/ubongo/router.py) owns config assembly — `plan_workflow()` combines `routing.yaml`, hysteresis, and the `/mode` override into a validated `WorkflowPlan`; `master.plan` maps it to a `Workflow` (adding persona model, resolved skill, and now the resolved verbosity level). The split is deliberate: router = config, master = turn state. [runner.py](src/ubongo/runner.py), the largest module, executes the workflow under one of six strategies (sequential, parallel, competitive, collaborative, debate, speculative), async inside, sync at its `execute()` boundary. [governance/](src/ubongo/governance/) scores risk, confidence, and reversibility independently and combines them through a priority-ordered rule matrix from `governance.yaml`; `require_approval` becomes an interactive y/n/why prompt ([governance/approval.py](src/ubongo/governance/approval.py)). Two newer governance modules sit beside it: [governance/grants.py](src/ubongo/governance/grants.py) (a grant-check rule placed *after* the safety rules) and [governance/verbosity.py](src/ubongo/governance/verbosity.py) (a per-domain length knob threaded into the composer persona as one directive line).

The agent fleet lives in [agents/](src/ubongo/agents/): Research, Coding, Evaluator, Critic, Execution, Repair, Memory, three Personas, and the Connector. An agent satisfies the `Agent` protocol and returns an `AgentResult`; non-composers return Findings threaded forward as `prior_findings`; exactly one agent per workflow carries `composer = True` and its output becomes the user-facing text (there is no composer module — folded into the runner and personas). Every LLM-calling agent goes through one **model-call envelope**, [agents/llm_run.py](src/ubongo/agents/llm_run.py), which itself calls [llm.py](src/ubongo/llm.py) (LiteLLM: single retry, token accounting, `before_llm`/`after_llm` events). Control signals flow down as a frozen typed `AgentDirectives` (now carrying `verbosity` alongside `skill`, `repair_prompt_hint`, etc.).

Durable state has one owner: the Memory Agent is the only writer (ADR-0002). The store, once a single 1,990-line module, is split into table-family modules behind the unchanged single-writer rule: [memory/store.py](src/ubongo/memory/store.py) (connection + per-turn core), [memory/trace.py](src/ubongo/memory/trace.py) (the trace tables + the `pending_approvals` record), and `evolution_state` / `authoring_state` / `index_state` / [grant_state.py](src/ubongo/memory/grant_state.py) / [jobs_state.py](src/ubongo/memory/jobs_state.py) for the subsystem rows. Recall blends a recency window with `sqlite-vec` semantic retrieval inside `store.recall()` (ADR-0010); [memory/vault_watch.py](src/ubongo/memory/vault_watch.py) ingests external Obsidian edits. Every outbound message — including synchronous CLI replies and now proactive job output — flows through [delivery/queue.py](src/ubongo/delivery/queue.py).

Four self-running loops share one lifecycle, [daemon.py](src/ubongo/daemon.py)'s `DaemonLoop` (thread, stop event, crash guard, budget/cron gate): the GP loop ([evolution/](src/ubongo/evolution/)) which mutates existing prompt/config targets and live-swaps approved variants via `active_evolutions` (ADR-0008); the authoring loop ([authoring/](src/ubongo/authoring/)) which drafts new skills into quarantine (ADR-0013); the vault watcher; and the **standing-jobs loop** ([jobs/](src/ubongo/jobs/)) — proactive output. All four boot paused. The jobs subsystem (`loop`, `runner`, `policy`, `delivery`, `commands`) runs config-defined jobs through `master.handle`, parks-and-raises on a missing grant, and enforces a quiet-hours hold + raise-TTL default-deny via the queue's existing `deliver_after`/`expires_at` filter.

External reach is one seam in each direction. Inbound, [mcp/server.py](src/ubongo/mcp/server.py) exposes `ubongo_send` over stdio/HTTP. Outbound, the Connector agent ([agents/connector.py](src/ubongo/agents/connector.py)) is the *only* door to external services, via [mcp/client.py](src/ubongo/mcp/client.py); any connector workflow is scored irreversible and reachable only by `/mode connector_session`. Shell execution is fenced entirely in [sandbox.py](src/ubongo/sandbox.py) — allowlist, no metacharacters, no traversal, empty child PATH, 10s timeout (ADR-0005).

The newest seam is **per-turn event streaming** ([web/console/stream_bridge.py](src/ubongo/web/console/stream_bridge.py)): `start_turn` runs `channel.run_turn` on a background thread and registers handlers on the named bus events ([events.py](src/ubongo/events.py)), forwarding a JSON-safe summary of each into a per-session queue; `event_stream` drains it as SSE frames. Single-flight (one active turn), so no event correlation is needed, and the console server starts no daemons so only the turn's own events fire. [web/console/app.py](src/ubongo/web/console/app.py) is the only module importing FastAPI/uvicorn. The stream observes the bus and resumes approvals; it never drives orchestration.

Extension points, in order of intent: new channels adapt over `channel.run_turn`; new behavior registers handlers on the named events in `events.py` (the console's streaming and the jobs' proactive policy are both this); new slash commands register in [commands.py](src/ubongo/commands.py); new tools default to CLI scripts behind constrained-bash; external services go through the Connector; evolvable behavior is addressed by target strings; approvals resolve the one `pending_approvals` record; an instance's identity is its data + config ([backup.py](src/ubongo/backup.py)).

## 3. Tech stack and dependencies

Python 3.11+ under uv. LiteLLM over OpenRouter for all model access, stdlib `sqlite3` plus `sqlite-vec` for storage and embeddings, PyYAML for config, `python-dotenv` for secrets. Optional, lazily-imported extras, each gating one channel: `streamlit` (web), the `mcp` SDK, `httpx` (Telegram), and `fastapi`+`uvicorn` (console). No orchestration framework, no Docker-in-app, no Redis — hand-rolled asyncio plus an event bus (ADR-0001).

Two dependencies do more structural work than they look. **LiteLLM** is the entire model-provider abstraction; every agent, the classifier, the evolution judge, and the Connector's planner funnel through one `complete()` in `llm.py`, so a LiteLLM change touches everything at once — and token-streaming (a future console phase) lives or dies on that one function. **`sqlite-vec`** keeps embeddings inside the canonical SQLite file, which is why "memory" is one single-writer component rather than a store plus a vector service. One non-code dependency is load-bearing for the trust posture: on Linux/Pi, **rootless Podman quadlets + an nftables egress allowlist** ([deploy/envelope/](deploy/envelope/)) make what leaves the machine enumerable below the model's discretion (ADR-0017) — zero `src/` change, and the thing that makes the cloud-relayed Telegram channel acceptable.

## 4. Key decisions and the constraints they created

Twenty-three accepted ADRs ([docs/adr/](docs/adr/)). The structurally consequential ones:

**Hand-rolled orchestration (0001)** traded ecosystem leverage for inspectability; any future orchestration complexity is paid in plain Python. **Single writer + universal queue (0002)** means no feature writes durable state or sends output directly — the constraint that made Telegram, standing jobs (proactive output), and the console all additive rather than structural. **The fixed pipeline (0003)** closed off per-channel turn handling; channels differ only in presentation. **Sandbox-in-code (0005)** closed off prompt-level safety and was load-bearing again for the Connector (0016), which *rejected* a CLI bridge for MCP and *deferred* a first-class tool layer. **Approved-not-autonomous self-modification (0006, 0013)** is a product identity: every loop boots paused, promotion/skill-approval/grant are human-only, and 0022 leans on this so a future *learned* verbosity target is safe by construction. **Live swap via one DB row (0008)** made evolution rollback trivial but couples three read paths to `active_evolutions`.

The trust-protocol ADRs each tightened an edge: the **outer envelope (0017)** put egress control below the app; the **resumable approval seam (0018)** made a held turn a persisted record with one re-issue path (approve from any channel); the **grant registry (0019)** added standing consent *after* the safety rules; **standing jobs (0021)** used the queue seam for proactive output with a quiet-hours/TTL default-deny posture for "no human at run time"; **contract/identity (0022)** made verbosity legible config and a backup just data (no secrets, grants re-arm); and **the live console (0023)** added per-turn event streaming as a handler on the bus, single-flight, LAN no-auth.

Baked-in tradeoffs: governance gates *delivery*, not *execution* — Connector/Execution side effects happen at execute time and the gate controls what reaches the user, with true pre-execution gating still future work (0016). The trust posture is single-user LAN/private-relay: no per-request auth or TLS anywhere in-app. And a full multi-agent governed turn is structurally *slow* — fine for text, a real obstacle for any future real-time voice channel.

## 5. The core invariant

**One governed seam per kind of consequence, with a human gate on self-modification and on first external reach.** Turns pass through `master.handle` via the channel core; durable writes through the Memory Agent; outbound messages through `notification_queue`; external calls through the Connector; shell through `sandbox.py`; approvals through the one `pending_approvals` record; and anything that changes the system itself — evolved prompts/config, authored skills — or reaches a new external capability class passes through quarantine/grant plus explicit approval. The console is the test case that proves the rule still holds: it is a rich new surface that only *observes* the bus and *resumes* approvals, opening no second path. A structural change that routes around any of these seams, or lets a loop promote its own output, violates the architecture's reason for existing.

## 6. Architectural debt and tension

**The runner is still the gravity well.** [runner.py](src/ubongo/runner.py) holds all six mode strategies, repair integration, and composer selection. It is the module most likely to fight a new execution mode or a streaming-aware turn.

**The named fracture was fixed; the budget pressure compounded.** The store split (v0.5 phase 02) cashed the one trigger-parked deepening candidate, but `src/` is now ~17,400 LOC, ~16% over the ~15,000 soft target. Each v0.5/v0.6 layer — jobs, verbosity, backup, the console — is a real subsystem, not a transport, and the project's "cut, don't expand" rule has only one scheduled clawback (retire Streamlit, v0.6 phase 05). The discipline that held early is now chronically strained.

**Designed core, accreted perimeter — and the perimeter is now most of the recent work.** Tiers 1–6 were specified and built to plan; everything since (web, authoring, profiler, two MCP halves, the trust quartet, and now a six-phase UI line) accreted. Each accretion was disciplined — additive channel, optional extra, ADR — but the *pattern* is the tension: the commit history is a steady pull toward platform capabilities, and the next addition that doesn't fit a seam is the standing risk.

**Enforcement keeps leaving the test suite.** The egress envelope, the Telegram relay, the console SSE, and any future voice path are all real boundaries pytest cannot see. "Green suite" certifies less of the actual trust/operational posture than it did at v0.1 — a widening gap between what the 1,057 tests prove and what the deployed system does.

**Fragile-but-load-bearing seams.** The `composer = True` attribute (a `getattr` on the agent class) decides what the user sees — invisible to type checkers. The live-swap reads guard against bootstrapping a DB from pure prompt assembly. The grant rule's *ordering* (after the safety rules) is correctness-critical and lives in rule sequence, not a type. The console's **single-flight** is the seam that will bend first if it ever needs concurrent turns or a "watch the GP loop" view (the `contextvar` correlation variant is named but deferred). And token-streaming, when it comes, must thread through the one `llm.complete()` chokepoint without regressing every other caller.
