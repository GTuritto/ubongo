# Ubongo — Architectural Snapshot

*2026-06-20, written from the code at v0.5.5 (branch `v0.5/04-telegram`, the Telegram channel merged on top of `main`). ~16,060 LOC under `src/ubongo/`, 20 accepted ADRs. Where docs and code disagree, the code is reported and the disagreement noted.*

## 1. Identity and purpose

Ubongo is a single-user, local-first AI mind: a multi-agent orchestration runtime and a self-improving system in one CLI-shaped package. A Master Agent runs every user turn through a fixed governed pipeline, dispatching disposable worker agents across six execution modes; a genetic-programming loop continuously evolves the system's own prompts and routing/tool-chain config, and a second loop drafts brand-new skills — but nothing the system produces about itself goes live without explicit human approval. The problem it solves is personal: one user (Giuseppe) wants an assistant whose behavior is inspectable, governable, and improvable, without ceding any of it to a SaaS or a framework. v0.1 (CLI) was certified complete; the system has since grown four additional channels (web, MCP server, and now Telegram) and an outbound MCP client, each added without touching the core.

## 2. Component map and data flow

Every turn, from any channel, takes the same path. The channel layer is presentation only:

```
repl.py / oneshot.py / web/app.py / mcp/server.py / telegram/bot.py   five fronts, thin
        └── channel.py            bootstrap() + run_turn(): the one turn envelope
              └── master.py       Master.handle(): classify → plan → execute → govern
                                  → compose → commit → enqueue
```

[channel.py](src/ubongo/channel.py) is the no-bypass rule made concrete (ADR-0002/0003): `bootstrap()` loads config + logging once and resolves the `UBONGO_PROFILE` knob; `run_turn()` optionally wraps the turn in cProfile, calls `master.handle` (resolved at call time so tests can patch it), and flushes the outbound queue. It starts no daemons — channels are the turn path only. Each front keeps only what differs: printing and exit codes (one-shot), rendering and Approve/Deny (web), TypedDict shaping and a worker-thread hop (MCP), long-poll + auth (Telegram), interactive prompts and per-session toggles (REPL).

Inside the pipeline: [classifier.py](src/ubongo/classifier.py) produces a tone/intent classification with a deterministic fallback (ADR-0009). [router.py](src/ubongo/router.py) owns config assembly — `plan_workflow()` combines `routing.yaml`, hysteresis, and the `/mode` override into a validated `WorkflowPlan`; `master.plan` maps it to a `Workflow` by adding turn state (persona model, resolved skill). The split is deliberate: router = config, master = turn state + registries. [runner.py](src/ubongo/runner.py) — at 1,121 lines the largest module — executes the workflow under one of six strategies (sequential, parallel, competitive, collaborative, debate, speculative), async inside, sync at its `execute()` boundary. [governance/](src/ubongo/governance/) scores risk, confidence, and reversibility independently and combines them through a priority-ordered rule matrix from `governance.yaml`; `require_approval` becomes an interactive y/n/why prompt ([governance/approval.py](src/ubongo/governance/approval.py)).

The agent fleet lives in [agents/](src/ubongo/agents/): Research, Coding, Evaluator, Critic, Execution, Repair, Memory, three Personas, and the Connector. The key abstractions: an agent satisfies the `Agent` interface and returns an `AgentResult`; non-composers return **Findings** threaded forward as `prior_findings`; exactly one agent per workflow carries `composer = True` and its output becomes the user-facing text (there is no composer module — the spec's `composer.py` was never built, folded into the runner and personas instead). Every LLM-calling agent goes through one **model-call envelope**, [agents/llm_run.py](src/ubongo/agents/llm_run.py), which owns timing, override resolution, error mapping, and logging, and itself calls [llm.py](src/ubongo/llm.py) (LiteLLM wrapper: single retry, token accounting, `before_llm`/`after_llm` events). Control signals flow down as a frozen typed `AgentDirectives`, not a string-keyed dict (ADR-0012).

Durable state is owned by one component: the Memory Agent is the only writer (ADR-0002). The store was a single 1,990-line module that the project itself flagged as the next fracture; v0.5 phase 02 split it into five table-family modules behind the unchanged single-writer rule: [memory/store.py](src/ubongo/memory/store.py) (now 592 lines — connection/bootstrap + the per-turn core: conversations, messages, summaries, sessions, recall), [memory/trace.py](src/ubongo/memory/trace.py) (the trace tables + view builders, plus the `pending_approvals` record), and [evolution_state.py](src/ubongo/memory/evolution_state.py) / [authoring_state.py](src/ubongo/memory/authoring_state.py) / [index_state.py](src/ubongo/memory/index_state.py) for the subsystem rows. Alongside sits the projected Markdown vault ([memory/vault.py](src/ubongo/memory/vault.py)) and the `sqlite-vec` embeddings ([memory/embeddings.py](src/ubongo/memory/embeddings.py)). Recall blends a recency window with semantic retrieval computed inside `store.recall()` — a documented deviation from the spec's event-handler placement, because the event fires too late (ADR-0010). [memory/vault_watch.py](src/ubongo/memory/vault_watch.py) polls for external Obsidian edits and ingests them, distinguishing its own writes by content hash, queuing conflicts rather than merging. Every outbound message — including synchronous CLI replies — flows through [delivery/queue.py](src/ubongo/delivery/queue.py).

Two self-modification subsystems mirror each other. [evolution/](src/ubongo/evolution/) mutates *existing* prompt and config targets: generation, sandboxed fitness evaluation against held-out conversations, lineage tracking, and promotion — approval writes one `active_evolutions` row, and the runtime read paths (`context.build_system_prompt`, `router.route_workflow`) consult it, so behavior live-swaps without a restart (ADR-0008). The `retry:repair` config target was *cut* in v0.5 (grant-registry Amendment 2) — its fitness was a structural proxy with no behavioral grounding, so it earned removal rather than improvement; the remaining evolvable targets are the three persona prompts plus `routing:default` and `toolchain:<workflow>`. [authoring/](src/ubongo/authoring/) drafts *new* skills into a quarantine directory invisible to the runtime, scores them side-effect-free, and only `/skill-candidates approve` materializes them (ADR-0013). Both background loops, plus the vault watcher, share one lifecycle: [daemon.py](src/ubongo/daemon.py)'s `DaemonLoop` owns the thread, stop event, crash guards, and budget/cron gating; each daemon keeps only its cycle work. All three boot paused.

External reach is one seam in each direction. Inbound, [mcp/server.py](src/ubongo/mcp/server.py) exposes `ubongo_send` over stdio/HTTP, calling the same `master.handle`; gated turns return `gated=true`. Outbound, the Connector agent ([agents/connector.py](src/ubongo/agents/connector.py)) is the *only* door to external services: it plans MCP tool calls with its model and executes them through [mcp/client.py](src/ubongo/mcp/client.py); any workflow containing it is scored irreversible, and it is reachable only by explicit `/mode connector_session`. Shell execution is fenced entirely in [sandbox.py](src/ubongo/sandbox.py) — allowlist, no metacharacters, no traversal, empty child PATH, 10s timeout — in code the LLM cannot rewrite (ADR-0005).

Two v0.5 additions close the asynchronous-approval gap that the prior snapshot named as the most predictable bend. The **approval exchange is now a typed, persisted, resumable seam** (ADR-0018): a held turn writes a `pending_approvals` record (in `trace.py`, keyed by `decision_id`); `Response.approval` is a typed `ApprovalRequest`; and `master.resume_approval(decision_id, choice)` is the single re-issue path. A turn gated in one channel can be approved in another — `/pending`, the web Approve button, `ubongo approve <id>`, or `/approve` over Telegram all resolve the same record. The **grant registry** (ADR-0019) makes consent persistent: a `grants` table ([memory/grant_state.py](src/ubongo/memory/grant_state.py)) holds standing consent for a capability class (`connector:<server>`), and a governance rule *after* the safety rules ([governance/grants.py](src/ubongo/governance/grants.py)) gates a connector turn's first encounter, then auto-proceeds once granted. Approving the first encounter writes the grant; `/grants` and `ubongo grants` manage them; revocation re-arms the ask and survives restart.

The extension points, in order of intent: new channels adapt over `channel.run_turn`; new v0.2+ behavior registers handlers on the named events in [events.py](src/ubongo/events.py) (a plain `dict[str, list[handler]]` bus with caught-and-logged handler failures); new slash commands register in [commands.py](src/ubongo/commands.py); new tools default to CLI scripts behind the constrained-bash skill; evolvable behavior is addressed by target strings; external services are reached only through the Connector. The Telegram channel ([telegram/](src/ubongo/telegram/)) is the proof the seams hold: `service.py` is the network-free core (auth via `allowed_user_ids` + the `/approve|/decline|/pending|/grants` router + turn handling over `channel.run_turn`), and `bot.py` is the only Bot-API module (httpx long-poll, lazy import, token in `.env`) — zero core change.

## 3. Tech stack and dependencies

Python 3.11+ under uv. The dependency surface is deliberately thin: LiteLLM (over OpenRouter) for all model access, stdlib `sqlite3` plus `sqlite-vec` for storage and embeddings, PyYAML for config, `python-dotenv` for secrets. Optional extras, all kept out of core deps and lazily imported: `streamlit` (web), the `mcp` SDK (server + client), and `httpx` (Telegram long-poll — no bot framework). There is no orchestration framework, no Docker in the app itself, no Redis — hand-rolled asyncio plus an event bus (ADR-0001).

Two dependencies do more structural work than they appear to. **LiteLLM** is the entire model-provider abstraction; every agent, the classifier, the evolution judge, and the Connector's planner all funnel through one `complete()` in `llm.py`, so a LiteLLM behavior change touches everything at once — but model swaps are one config edit (`config/settings.yaml::models`). **`sqlite-vec`** keeps embeddings inside the same SQLite file as the canonical record, which is why "memory" can be one single-writer component instead of a store plus a vector service; it is loaded lazily and every path degrades to recency-only recall if it is absent (ADR-0010).

One non-code dependency is now load-bearing for the deployment posture: on the Linux/Pi target, **rootless Podman quadlets + an nftables egress allowlist** ([deploy/envelope/](deploy/envelope/)) enumerate what may leave the machine in `/etc/ubongo/egress.hosts` and enforce it below the model's discretion (ADR-0017). This is zero `src/` change and Linux-only by design — the macOS dev machine is not enveloped — but it is what makes the cloud-relayed Telegram channel acceptable.

## 4. Key decisions and the constraints they created

Twenty accepted ADRs in [docs/adr/](docs/adr/). The structurally consequential ones:

**Hand-rolled orchestration (0001)** traded ecosystem leverage for full inspectability; any future orchestration complexity must be paid for in plain Python. **Single writer + universal queue (0002)** means no feature may write durable state or send output directly — the constraint that made Telegram (v0.5) and proactive jobs (v0.3, not yet wired) additive rather than structural. **The fixed pipeline (0003)** closed off per-channel turn handling; channels can only differ in presentation. **Sandbox-in-code (0005)** closed off prompt-level safety and was load-bearing again for the Connector decision (0016), which *rejected* a CLI bridge for MCP because it would carve a network hole into the sandbox and *deferred* a first-class tool layer because it would scatter external side effects and dissolve the single governance point. **Approved-not-autonomous self-modification (0006, 0013)** is a product-identity choice: both loops boot paused, promotion and skill approval are human-only, and the sandbox allowlist is a human-only change. **Live swap via one DB row (0008)** made evolution rollback trivial but couples three read paths to the `active_evolutions` table. **Additive channels over one core (0015, candidate 14)** turned the no-bypass rule from a convention into a function — `channel.run_turn`.

The three newest ADRs each tightened a previously soft edge. **The resumable approval seam (0018)** converted approval from an in-channel y/n into a persisted record with one re-issue path, which is the prerequisite for approve-later and the thing that makes Telegram approvals possible at all. **The grant registry (0019)** added a consent-caching layer *after* the safety rules — standing consent that survives restart, server-granular (per-tool deferred to a future revision of 0016), and it explicitly cut the weak `retry:repair` evolvable target rather than carry dead weight. **The outer envelope (0017)** put deployment-level egress control below the application, which is what let the cloud-relayed Telegram channel (0020) ship without weakening the trust posture.

The baked-in tradeoffs that remain: governance gates *delivery*, not *execution* — Execution and Connector side effects happen at execute time and the gate controls what reaches the user, with true pre-execution gating of individual calls still future work (0016). And the trust posture is single-user LAN/private-relay: no per-request auth or TLS in the app; Telegram auth is a flat `allowed_user_ids` allowlist (empty = deny all), and the egress envelope, not the app, is the network boundary.

## 5. The core invariant

**One governed seam per kind of consequence, with a human gate on self-modification and on first external reach.** Turns pass through `master.handle` via the channel core; durable writes pass through the Memory Agent; outbound messages pass through `notification_queue`; external calls pass through the Connector; shell execution passes through `sandbox.py`; approval flows through the one resumable `pending_approvals` record; and anything that changes the system itself — evolved prompts, evolved config, authored skills — or reaches a new external capability class passes through quarantine/grant plus explicit approval. A structural change that introduces a second path around any of these seams, or that lets a loop promote its own output, violates the architecture's reason for existing.

## 6. Architectural debt and tension

**The runner is the gravity well.** [runner.py](src/ubongo/runner.py) at 1,121 lines holds all six mode strategies, repair integration, and the composer selection. It grew slightly since the store split relieved its sibling. Peer-replacement repair is wired into all five fan-out modes with per-mode tests; the spec for it lives at `openspec/specs/fanout-peer-replacement/spec.md` and the *change* proposal was reconciled (archived) in v0.5 phase 00 — the stale-ledger trap that misled the prior snapshot's first draft has been closed.

**The named fracture was repaired; the budget pressure was not.** `store.py` fell from 1,990 to 592 lines when v0.5 phase 02 split it into five table-family modules — the deepening candidate the project had trigger-parked. But total `src/` LOC is now ~16,060, past the ~15,000 soft target by ~7%. The project's own rule is "cut, don't expand the budget," which keeps every new subsystem proposal in tension with itself; the discipline so far has been to add *transports* (Telegram is ~240 lines) rather than subsystems.

**Designed core, accreted perimeter.** Tiers 1–6 were specified and built to plan; everything since v0.1 (web, authoring, profiler, two MCP halves, the approval/grant/envelope/Telegram quartet) accreted against a spec that listed some of it as out of scope. Each accretion was disciplined — additive channel, optional extra, ADR — but the *pattern* is the tension: the original sequencing said one thing, the commit history shows a steady pull toward platform capabilities. The discipline has held precisely because the seams existed; the standing risk is the first addition that doesn't fit a seam.

**Asynchronous approval is solved structurally but assumes a reachable human.** The resumable seam (0018) removed the old "approval assumes a human is present in *this* channel" debt — a turn gated in REPL can now be approved from a phone. What remains is that *someone* must eventually answer: a held turn occupies a `pending_approvals` row indefinitely, and there is no expiry, escalation, or quiet-hours policy yet (deferred to the v0.2/v0.3 notification-policy work). For proactive output (v0.3), where there may be no human in any loop at send time, this will need a default-deny or auto-expire posture that does not exist today.

**Fragile-but-load-bearing seams.** The `composer = True` attribute convention (a `getattr` on the agent class) decides what the user sees — invisible to type checkers, easy to violate silently. The live-swap reads guard against bootstrapping a DB from pure prompt assembly, a subtle invariant tested but easy to regress. The grant rule's *ordering* — after the safety rules, so a grant can never auto-proceed something the matrix would reject — is correctness-critical and lives in rule sequence, not in a type. And the egress envelope is enforcement that lives entirely outside `src/` and outside the test suite, so a change to what the Connector or a CLI script reaches can pass every pytest and still be silently blocked (or silently allowed) by `/etc/ubongo/egress.hosts` on the deployed box.
