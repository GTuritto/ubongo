# Ubongo — Architectural Snapshot

*2026-06-12, written from the code at v0.1.5 (`main`). 15,396 LOC under `src/ubongo/`, 960 tests green. Where docs and code disagree, the code is reported and the disagreement noted.*

## 1. Identity and purpose

Ubongo is a single-user, local-first AI mind: a multi-agent orchestration runtime and a self-improving system in one CLI-shaped package. A Master Agent runs every user turn through a fixed governed pipeline, dispatching disposable worker agents across six execution modes; a genetic-programming loop continuously evolves the system's own prompts and routing/tool-chain/retry config, and a second loop drafts brand-new skills — but nothing the system produces about itself goes live without explicit human approval. The problem it solves is personal: one user (Giuseppe) wants an assistant whose behavior is inspectable, governable, and improvable, without ceding any of it to a SaaS or a framework.

## 2. Component map and data flow

Every turn, from any channel, takes the same path. The channel layer is presentation only:

```
repl.py / oneshot.py / web/app.py / mcp/server.py        four fronts, thin
        └── channel.py            bootstrap() + run_turn(): the one turn envelope
              └── master.py       Master.handle(): classify → plan → execute → govern
                                  → compose → enqueue → memory commit
```

Inside the pipeline: [classifier.py](src/ubongo/classifier.py) produces a tone/intent classification with a deterministic fallback (ADR-0009). [router.py](src/ubongo/router.py) owns config assembly — `plan_workflow()` combines `routing.yaml`, hysteresis, and the `/mode` override into a validated `WorkflowPlan`; `master.plan` maps it to a `Workflow` by adding turn state (persona model, resolved skill). The split is deliberate: router = config, master = turn state + registries. [runner.py](src/ubongo/runner.py) — at 1,115 lines the largest module — executes the workflow under one of six strategies (sequential, parallel, competitive, collaborative, debate, speculative), async inside, sync at its `execute()` boundary. [governance/](src/ubongo/governance/) scores risk, confidence, and reversibility independently and combines them through a priority-ordered rule matrix from `governance.yaml`; `require_approval` becomes an interactive y/n/why prompt ([governance/approval.py](src/ubongo/governance/approval.py)).

The agent fleet lives in [agents/](src/ubongo/agents/): Research, Coding, Evaluator, Critic, Execution, Repair, Memory, three Personas, and the Connector. The key abstractions: an agent satisfies the `Agent` interface and returns an `AgentResult`; non-composers return **Findings** threaded forward as `prior_findings`; exactly one agent per workflow carries `composer = True` and its output becomes the user-facing text (there is no composer module — the spec's `composer.py` was never built, folded into the runner and personas instead). Every LLM-calling agent goes through one **model-call envelope**, [agents/llm_run.py](src/ubongo/agents/llm_run.py), which owns timing, override resolution, error mapping, and logging, and itself calls [llm.py](src/ubongo/llm.py) (LiteLLM wrapper: single retry, token accounting, `before_llm`/`after_llm` events). Control signals flow down as a frozen typed `AgentDirectives`, not a string-keyed dict (ADR-0012).

Durable state is owned by one component: the Memory Agent is the only writer to [memory/store.py](src/ubongo/memory/store.py) (stdlib SQLite, 1,990 lines — the second-largest module and a named refactor trigger), the projected Markdown vault ([memory/vault.py](src/ubongo/memory/vault.py)), and the `sqlite-vec` embeddings ([memory/embeddings.py](src/ubongo/memory/embeddings.py)). Recall blends a recency window with semantic retrieval computed inside `store.recall()` — a documented deviation from the spec's event-handler placement, because the event fires too late (ADR-0010). [memory/vault_watch.py](src/ubongo/memory/vault_watch.py) polls for external Obsidian edits and ingests them, distinguishing its own writes by content hash, queuing conflicts rather than merging. Every outbound message — including synchronous CLI replies — flows through [delivery/queue.py](src/ubongo/delivery/queue.py).

Two self-modification subsystems mirror each other. [evolution/](src/ubongo/evolution/) mutates *existing* prompt and config targets: generation, sandboxed fitness evaluation against held-out conversations, lineage tracking, and promotion — approval writes one `active_evolutions` row, and the runtime read paths (`context.build_system_prompt`, `router.route_workflow`) consult it, so behavior live-swaps without a restart (ADR-0008). [authoring/](src/ubongo/authoring/) drafts *new* skills into a quarantine directory invisible to the runtime, scores them side-effect-free, and only `/skill-candidates approve` materializes them (ADR-0013). Both background loops, plus the vault watcher, share one lifecycle: [daemon.py](src/ubongo/daemon.py)'s `DaemonLoop` owns the thread, stop event, crash guards, and budget/cron gating; each daemon keeps only its cycle work. All three boot paused.

External reach is one seam in each direction. Inbound, [mcp/server.py](src/ubongo/mcp/server.py) exposes `ubongo_send` over stdio/HTTP, calling the same `master.handle`; gated turns return `gated=true` and are never approvable over MCP. Outbound, the Connector agent ([agents/connector.py](src/ubongo/agents/connector.py)) is the *only* door to external services: it plans MCP tool calls with its model and executes them through [mcp/client.py](src/ubongo/mcp/client.py); any workflow containing it is scored irreversible, and it is reachable only by explicit `/mode connector_session`. Shell execution is fenced entirely in [sandbox.py](src/ubongo/sandbox.py) — allowlist, no metacharacters, no traversal, empty child PATH, 10s timeout — in code the LLM cannot rewrite (ADR-0005).

The extension points, in order of intent: new channels adapt over `channel.run_turn`; new v0.2+ behavior registers handlers on the named events in [events.py](src/ubongo/events.py); new slash commands register in [commands.py](src/ubongo/commands.py); new tools default to CLI scripts behind the constrained-bash skill; evolvable behavior is addressed by target strings.

## 3. Tech stack and dependencies

Python 3.11+ under uv. The dependency surface is deliberately thin: LiteLLM (over OpenRouter) for all model access, stdlib `sqlite3` plus `sqlite-vec` for storage and embeddings, PyYAML for config, `.env` for secrets. Optional extras, kept out of core: `streamlit` (web channel) and the `mcp` SDK (imported lazily, only in `mcp/server.py` and `mcp/client.py`). There is no orchestration framework, no Docker, no Redis — hand-rolled asyncio plus an event bus (ADR-0001).

Two dependencies do more structural work than they appear to. **LiteLLM** is the entire model-provider abstraction; every agent, the classifier, the evolution judge, and the Connector's planner all funnel through one `complete()` in `llm.py`, so a LiteLLM behavior change touches everything at once — but also means model swaps are one config edit. **`sqlite-vec`** keeps embeddings inside the same SQLite file as the canonical record, which is why "memory" can be one single-writer component instead of a store plus a vector service; it is loaded lazily and every path degrades to recency-only recall if it is absent (ADR-0010).

## 4. Key decisions and the constraints they created

Sixteen accepted ADRs in [docs/adr/](docs/adr/) (STATE.md still says twelve — doc rot). The structurally consequential ones:

**Hand-rolled orchestration (0001)** traded ecosystem leverage for full inspectability; any future orchestration complexity (cross-session workflows, distributed agents) must be paid for in plain Python. **Single writer + universal queue (0002)** means no feature may write durable state or send output directly — the constraint that makes Telegram (v0.2) and proactive jobs (v0.3) additive rather than structural. **The fixed pipeline (0003)** closed off per-channel turn handling; channels can only differ in presentation. **Sandbox-in-code (0005)** closed off prompt-level safety and was load-bearing again in 2026: the Connector decision (0016) *rejected* a CLI bridge for MCP precisely because it would carve a network hole into the sandbox, and *deferred* a first-class tool layer because it would scatter external side effects across the fleet and dissolve the single governance point. **Approved-not-autonomous self-modification (0006, 0013)** is a product identity choice as much as a technical one: both loops boot paused, promotion and skill approval are human-only, and the sandbox allowlist is a human-only change. **Live swap via one DB row (0008)** made evolution rollback trivial but couples three read paths to the `active_evolutions` table. **Additive channels over one core (0015, candidate 14)** turned the no-bypass rule from a convention into a function — `channel.run_turn` — which is now the cheapest place to add Telegram and the most dangerous place to regress.

The baked-in tradeoffs: governance gates *delivery*, not *execution* — Execution and Connector side effects happen at execute time and the gate controls what reaches the user, with true pre-execution gating of individual calls explicitly future work (0016). And the trust posture is single-user LAN: no auth, no TLS, anywhere.

## 5. The core invariant

**One governed seam per kind of consequence, with a human gate on self-modification.** Turns pass through `master.handle` via the channel core; durable writes pass through the Memory Agent; outbound messages pass through `notification_queue`; external calls pass through the Connector; shell execution passes through `sandbox.py`; and anything that changes the system itself — evolved prompts, evolved config, authored skills — passes through quarantine plus explicit approval. A structural change that introduces a second path around any of these seams, or that lets a loop promote its own output, violates the architecture's reason for existing.

## 6. Architectural debt and tension

**The runner is the gravity well.** [runner.py](src/ubongo/runner.py) at 1,115 lines holds all six mode strategies, repair integration, and the composer selection. Two deepening candidates aimed at it (17 merged, 09 judged speculative and skipped). Peer-replacement repair is in fact wired into all five fan-out modes with per-mode tests ([runner.py:568](src/ubongo/runner.py#L568), 636, 753, 881, 1009) — but the repo's own change ledger says otherwise: `openspec/changes/complete-fanout-peer-replacement/` proposes this exact work and was never archived, a stale claim capable of misleading reviews (it misled this one's first draft).

**`store.py` is the named next fracture.** At 1,990 lines it is the module the project itself flagged: deepening candidate 19 (split it) is trigger-parked on its continued growth, and STATE.md doubles that trigger as the LOC shrink review. The soft budget (~15,000 LOC) has been crossed for the first time; the project's own rule is "cut, don't expand the budget," which puts every new subsystem proposal in tension with itself.

**Designed core, accreted perimeter.** Tiers 1–6 were specified and built to plan; the post-v0.1 layer (web, authoring, profiler, two MCP halves) accreted against a spec that listed some of it as out of scope. Each accretion was disciplined — additive channel, optional extra, ADR — but the *pattern* is the tension: the spec's sequencing says Telegram next, the commit history says the pull is toward platform capabilities. The discipline has held so far precisely because the seams existed; the risk is the first addition that doesn't fit a seam.

**Approval assumes a human is present.** The interactive y/n/why gate works for REPL and web. Over MCP, gated turns stall by design; for v0.2 Telegram and v0.3 proactive jobs, an *asynchronous* approval surface will be needed, and nothing in the current gate is built for approval-later. This is the most predictable place the architecture will be forced to bend.

**Fragile-but-load-bearing seams.** The `composer = True` attribute convention (a `getattr` on the agent class) decides what the user sees — invisible to type checkers, easy to violate silently. The live-swap reads guard against bootstrapping a DB from pure prompt assembly, a subtle invariant tested but easy to regress. And the `retry:repair` fitness signal is a structural proxy with no behavioral grounding — arguably a target to delete rather than improve.
