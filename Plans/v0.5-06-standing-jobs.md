# v0.5 Phase 06 — Standing jobs (proactive output)

Phase 06 of [Plans/v0.5-trust-protocol.md](v0.5-trust-protocol.md). Branch
`v0.5/06-standing-jobs` → fills the **v0.5.6** slot (CI derives the version from the branch name;
do not hand-edit VERSION). The first time Ubongo speaks **unprompted**: a scheduled job composes a
turn, runs it through the one orchestration seam, and delivers the result through the existing
queue. It is the v0.3 "proactive output" the architecture reserved a seam for (flow starts at
`enqueue`), built additively on the four seams the trust protocol already shipped: the daemon
lifecycle (candidate 15), the notification queue (ADR-0002), the resumable approval seam
(ADR-0018), and the grant registry (ADR-0019).

## The shape

A standing job is a scheduled, governed turn that the user does not type. Nothing about
orchestration changes: a job builds a message, calls `channel.run_turn` (→ `master.handle`), and
its output flows through `notification_queue`. The new parts are scheduling, a definition-time
grant bundle, and a proactive-delivery policy.

- **`StandingJobsLoop`** — a **fourth `DaemonLoop`** (joining the GP loop, the authoring loop, and
  the vault watcher), booting **paused** like its siblings, throttled by the same rolling-hour
  budget + cron pacing. Each cycle picks the due job, runs its turn, and narrates completion in
  **one line** through the queue. Controlled by `/jobs status|pause|resume|off`.
- **`memory/jobs_state.py`** — a new table-family module (the store-split pattern from Phase 02):
  `standing_jobs` (id, name, schedule, grant-bundle classes, prompt template, status, last_run,
  next_run) and `job_runs` (per-cycle audit: outcome ∈ `delivered | parked | held | skipped`,
  the `decision_id` when parked, timings). The Memory Agent remains the only writer.
- **Definition-time grant bundle** — a job declares its capability needs *when it is registered*
  (its `connector:<server>` classes). The bundle is approved **once** through the Phase-03 seam,
  which writes the Phase-05 grants. Thereafter the job's runs proceed **within the bundle**
  without asking. A run that reaches for anything **outside** the bundle gates as usual, which
  triggers park-and-raise.
- **Park-and-raise** — when a job's turn returns `require_approval`, the job **parks** (status
  `parked`, the `pending_approvals` record persists via ADR-0018) and **raises itself** over the
  proactive channel ("Job *news-digest* needs approval for X — `/approve <id>`"). Approving later
  (from the phone, ADR-0018/0020) resumes it on the next cycle. This is how a job with no human
  present at run time still makes forward progress.
- **Proactive delivery policy** — proactive sends pass through a `before_send` handler (the
  minimal first cut of the notification-policy engine deferred from v0.2): a **quiet-hours**
  window holds sends and **batches** them into a catch-up; a parked raise carries a **TTL** and
  **auto-declines on expiry** (default-deny) so nothing waits forever and nothing fires at 3am.

```text
StandingJobsLoop (4th DaemonLoop, boots paused)
   │ due job → build message
   ▼
channel.run_turn → master.handle   (no bypass: classified, governed, persisted)
   │
   ├── auto / within-bundle grant → queue → before_send policy
   │        ├── inside quiet hours → HOLD (batch into catch-up)
   │        └── else → DELIVER (Telegram if configured; else REPL catch-up) + 1-line narration
   └── require_approval (outside bundle) → PARK + raise over the proactive channel
            └── approve-later (ADR-0018) → resumes next cycle  |  TTL expiry → auto-decline, retry next schedule
```

The **news digest** is the first job and the acceptance test: a daily job that fetches headlines
through the Connector (the one external door, ADR-0016), composes a digest, and delivers it
proactively — parking gracefully if its news grant is missing.

## What changes

1. `memory/jobs_state.py` (the two tables) and the `StandingJobsLoop` daemon; `/jobs` REPL
   command + `ubongo jobs` CLI surface (list / status / pause / resume / run-now).
2. **`config/jobs.yaml`** — jobs are **config-defined** for v1 (one user, few stable jobs; lighter
   than a CRUD command and LOC-honest). Each entry: `name`, `schedule` (cron-ish), `grant_bundle`
   (capability classes), `prompt`. Registering a new job with an un-granted bundle asks once
   through the seam on first run.
3. A **`before_send` proactive-policy handler** wired on the named event: quiet-hours hold +
   batch, and parked-raise TTL / auto-decline. Default config is permissive except the quiet-hours
   window. Typed turns (REPL/one-shot/web/MCP/Telegram) are untouched — the policy only inspects
   `source ∈ {proactive, job}`.
4. **ADR-0021** records proactive output: the fourth daemon, the definition-time grant bundle, the
   park-and-raise loop, and the approval-expiry posture (quiet hours + TTL default-deny) that
   makes "no human at run time" safe.
5. `UBONGO_DISABLE_JOBS` off-switch (suite stays daemon-free, mirroring the other loops). The job
   daemon is enabled in config, boots paused, and is started/stopped by the REPL alongside the
   others.
6. The news-digest job ships as the worked example in `config/jobs.yaml` (disabled by default
   until its Connector server + grant exist).

## Decisions to settle before code (the point of this draft)

These are the design choices that determine whether proactive output is safe; confirm or redirect.

- **D1 — Job definition surface.** *Recommend config-driven `config/jobs.yaml`* for v1 over a
  `/jobs add` CRUD command: fewer moving parts, no live-editing of scheduled side effects, LOC-light
  (the budget is ~7% over). `/jobs` stays read/control only (status/pause/resume/run-now).
- **D2 — The approval-expiry posture (the gate I flagged).** *Recommend* three controls together:
  (a) **quiet hours** in config hold proactive sends and batch them into a catch-up on the next
  open window or next REPL launch; (b) a parked raise carries a **TTL** (`jobs.raise_ttl_hours`,
  default 24) and **auto-declines on expiry** — default-deny, the job retries on its next schedule,
  never piling up; (c) definition-time grant bundles keep raises rare. This is the "default-deny or
  auto-expire" posture the state briefing said 06 could not ship without.
- **D3 — External reach for the news digest.** *Recommend the Connector* (ADR-0016, the single
  external door) over a constrained-bash CLI script — the sandbox blocks network by design, and the
  Connector already carries the grant-bundle semantics. The digest's bundle is `connector:<news>`.
- **D4 — Proactive transport.** *Recommend* the queue + `before_send` policy, delivering to
  Telegram when configured (the natural remote surface, Phase 04) and otherwise holding for a REPL
  **catch-up** on next launch. No new transport is built; proactive output reuses existing ones.

## QA test plan

### Acceptance criteria (exit = all checked)

- [ ] **AC-1 No-bypass.** A job's turn goes through `channel.run_turn` → `master.handle`: classified,
      governed, persisted (a `workflow_runs` row, a `job_runs` row, a queue row) exactly like a typed
      turn. The daemon never composes or delivers around the seam.
- [ ] **AC-2 Boots paused.** With `jobs.enabled: true`, the loop starts **paused** (persisted
      status); nothing runs until `/jobs resume`. `UBONGO_DISABLE_JOBS=1` keeps the thread from
      starting; the suite is daemon-free.
- [ ] **AC-3 Runs within the bundle unattended.** A job whose `grant_bundle` is granted runs on
      schedule, delivers, and narrates completion in **one line** — no prompt, no human.
- [ ] **AC-4 Parks on a missing grant.** A job reaching outside its bundle gates: `job_runs.outcome
      = parked`, a `pending_approvals` record exists, and a raise is enqueued naming the
      `decision_id`. The job does **not** deliver its result.
- [ ] **AC-5 Approve-later resumes it.** Approving the parked raise (via `/approve <id>` /
      `ubongo approve` / Telegram, the ADR-0018 seam) lets the job complete on its next cycle; the
      record flips to `approved`.
- [ ] **AC-6 Quiet hours hold + batch.** A proactive send inside the quiet-hours window is **held**
      (`job_runs.outcome = held`), not delivered, and surfaces as a catch-up at the next open window
      / REPL launch. A typed turn in the same window is unaffected.
- [ ] **AC-7 Raise TTL auto-declines.** A parked raise past `raise_ttl_hours` **auto-declines**
      (default-deny); the `pending_approvals` record flips to `declined`/`expired`, the job logs
      `skipped`, and it retries on its next schedule — no unbounded backlog.
- [ ] **AC-8 Safety still wins.** A destructive job turn gates by the safety rules (rule 1), never
      via the grant bundle; the bundle never auto-proceeds something the matrix would reject.
- [ ] **AC-9 Additive.** REPL / one-shot / web / MCP / Telegram and all orchestration are
      byte-unchanged for typed turns; the `before_send` policy only acts on `source ∈ {proactive,
      job}`; no new core dependency.
- [ ] **AC-10 Ops surfaces.** `/jobs status|pause|resume|off` and `ubongo jobs [run <name>]` work;
      `config/jobs.yaml` parse errors fail friendly (no traceback); the news-digest example is
      present and disabled by default.
- [ ] **AC-11 News digest end to end (live, needs a Connector news server).** The daily job fetches
      via the Connector, composes a digest, delivers it proactively, and parks if its grant is
      revoked. (Pi/config-dependent, like the other live external checks.)
- [ ] **AC-12 Full suite green** with `tests/test_jobs_state.py` + `tests/test_standing_jobs.py` +
      the `before_send` policy tests; ADR-0021 + CHANGELOG `## v0.5.6` shipped (version CI-derived).

### Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| A job spams the user / fires at 3am | trust erosion, the core fear of proactive output | quiet-hours hold + batch (AC-6); one-line narration; the loop is rate-throttled and boots paused (AC-2) |
| A parked raise waits forever | backlog, stale approvals | raise TTL auto-declines, job retries next schedule (AC-7) — the default-deny posture |
| The grant bundle silently widens a job's reach | a job does more than approved | bundle is declared at definition and approved once; anything outside it re-gates (AC-4); safety rules still win (AC-8) |
| Proactive send bypasses governance | an ungoverned message reaches the user | jobs run through `master.handle`; the queue + `before_send` are the only delivery path (AC-1/AC-9) |
| The 4th daemon destabilizes the others | background instability | it subclasses the shared `DaemonLoop` (crash-swallow, budget/cron gating already proven on three loops); `UBONGO_DISABLE_JOBS` keeps the suite clean (AC-2) |
| Subsystem grows the LOC budget (already ~7% over) | budget breached further | reuse DaemonLoop/queue/grants/approval wholesale; config-driven jobs over a CRUD command (D1); the new code is the loop + one table module + one policy handler |
| News fetch needs network the sandbox blocks | the digest can't reach out | route external reach through the Connector (D3), not a CLI script — the governed external door, already grant-aware |

## Why this fits the architecture

It adds no second path around any governed seam. A job's turn still goes through `master.handle`;
its output still goes through `notification_queue`; its consent still goes through the grant
registry; its approvals still resolve the one `pending_approvals` record; its external reach still
goes through the Connector. The new primitives — a fourth daemon, a job table, and a `before_send`
proactive policy — are all the shapes the project already prescribes (a `DaemonLoop` subclass, a
table-family module, an event handler). Proactive output is the seam finally being used, not a new
seam being cut.
