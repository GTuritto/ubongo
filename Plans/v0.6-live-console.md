# v0.6 — The live console (the streaming web channel)

A new minor line, themed on *seeing the machine think*. Where v0.5 (the trust protocol) hardened
what Ubongo is allowed to do, v0.6 makes a turn legible while it runs: a browser console that
shows the agent roster for the turn (who is working now, who is queued, who is retiring), streams
the pipeline activity step by step, and renders the answer as it arrives. It is a sixth channel,
additive over the one turn seam, and it reuses data Ubongo already produces. Branches follow the
house convention `v0.6/NN-<name>` → `0.6.NN` (CI derives the version from the branch name; do not
hand-edit VERSION).

**Scope note on security.** This is an internal, single-user LAN tool. No auth, no TLS, no
per-request principals. The console binds to the LAN and is trusted like the existing Streamlit
page. That is a deliberate non-goal, recorded so it is a decision and not an oversight. If the box
is ever exposed, the Phase-01 egress envelope and a future auth layer are the controls, not this
plan.

## The shape

A channel is a presentation layer over the one turn envelope (`channel.run_turn` →
`master.handle` → queue flush). The console is an additive adapter over it, the same pattern as
MCP and Telegram. The difference is liveness: the other channels run a turn and return one
response; the console wants to show the turn *as it happens*. That requires exactly one new
mechanism, and Ubongo is already built for it.

- **The data already exists.** The router computes a `WorkflowPlan` (the full agent tuple, the
  execution mode, the round count) *before* any agent runs, so the roster is known up front. The
  event bus (`events.py`) already dispatches the pipeline beats in order: `before_classify`,
  `after_classify`, `before_plan`, `after_plan`, `before_execute`, `agent_started`,
  `agent_completed`, `agent_failed`, `before_govern`, `after_govern`, `before_compose`,
  `after_compose`, `before_send`, `after_send`. The trace tables (`workflow_runs`, `agent_runs`)
  already persist the same beats for history. Recall already returns the memories a turn drew on
  (`RecallContext.semantic_messages` + the vault-link graph).
- **The one new mechanism: per-turn event streaming.** Today a turn runs synchronously and
  returns once. To stream, the console runs the turn in a background task and forwards bus events
  to the browser over Server-Sent Events (SSE) as they fire. This is a registered event handler
  pushing to a queue that the SSE endpoint drains. It is precisely the "new behavior ships as
  event handlers, not pipeline edits" rule from CLAUDE.md. No change to `master.handle`, the
  runner, or the agents.

```text
browser (console.html + a little JS)
   │   POST /turn  (message, persona)        ── starts a turn, returns a stream_id
   │   GET  /stream/<stream_id>  (SSE)        ── live events for that turn
   ▼
web/console/app.py  (FastAPI)
   ├── runs channel.run_turn(...) in a background task
   ├── stream_bridge.py: event handlers (scoped to stream_id) → asyncio.Queue → SSE frames
   └── reads WorkflowPlan/agent_runs for roster, RecallContext for sources,
       pending_approvals/resume_approval for the gate
              │ (no bypass)
              ▼
        channel.run_turn → master.handle → queue flush
```

### The correlation problem (the one real design decision)

The bus is global, synchronous, and in-process. The evolution and authoring daemons also dispatch
agent events during sandbox evaluation, and two browser tabs could each start a turn. A naive
global handler would mix events from all of them into one stream. Two options:

1. **Single-flight v1 (trivial).** The console serves one active turn at a time; a second `POST
   /turn` is rejected while one is live. Correct for a single user at one keyboard, zero new
   plumbing. Daemon events are filtered out by event name (the console only forwards the
   turn-pipeline events, and the daemons' cycle work is distinguishable).
2. **Correlated v2 (clean).** Thread a per-turn `stream_id` through a `contextvar` set at the top
   of `channel.run_turn`; the forwarder tags each event with the current `stream_id` and routes
   it to the matching queue. This makes concurrent turns and a "watch the GP loop" view possible
   later.

Recommendation: ship v1 single-flight in Phase 00, add the `contextvar` correlation in Phase 01
only if the roster work needs it. Do not build v2 speculatively.

## Phase breakdown

Each phase is its own branch and draft PR, merged by you, version `0.6.NN`.

- **Phase 00 — The streaming seam.** `web/console/app.py` (FastAPI) + `stream_bridge.py` (the
  event-forwarder + background-task runner + SSE endpoint) + a bare `console.html` that prints the
  raw event log. Single-flight. Optional extra `[console]` (fastapi + uvicorn + sse-starlette or
  hand-rolled SSE), lazily imported, kept out of core like `streamlit`/`mcp`/`telegram`.
  `ubongo console` entrypoint, ctl + systemd unit + `start-ubongo-console.sh`, installer support.
  **ADR-0021** records the per-turn event-streaming seam and the explicit no-auth LAN posture.
  Exit: a typed turn in the browser streams its pipeline beats live, fully through `master.handle`.
- **Phase 01 — The agent roster panel.** The real UI. Roster initialized from the `after_plan`
  event (agent tuple + mode + rounds); each agent flips pending → running → done/failed off
  `agent_started`/`agent_completed`/`agent_failed`. "Retiring in the next rounds" = agents behind
  the run cursor. Peer-replacement renders as the original agent retiring and its peer joining
  (`agent_failed` → a fresh `agent_started`). This is the headline feature and the part Ubongo is
  best positioned to do, because the lineup is planned, not guessed.
- **Phase 02 — The activity stream + response render.** The ordered step log ("Classifying →
  Planning → Research running → Evaluator scoring → Governance → Composing") from the named events,
  and the final composed answer rendered when `after_compose`/`after_send` lands. A "Step N/M"
  progress read off the roster length.
- **Phase 03 — Approval and sources in the console.** The governance gate as a button: when a turn
  returns `response.approval`, the console shows Approve/Decline wired to `master.resume_approval`
  (the Phase-03 seam), and a `/pending` list. A "Sources" panel rendering
  `RecallContext.semantic_messages` and the vault links the turn used. No new orchestration; both
  read existing data.
- **Phase 04 (optional) — Token-streaming the answer.** The only part that touches the core:
  add streaming support to `llm.complete()` (the single model chokepoint, `llm.py`) and stream the
  composer agent's tokens to the console. Marked optional because the screenshots are activity
  streaming, not token streaming, and this is the one invasive change. Behind a flag; the
  non-streaming path stays the default and is byte-unchanged.
- **Phase 05 — Retire Streamlit.** Once the console covers the chat surface, remove `web/` (the
  Streamlit channel) and its extra, so there is one web UI, not two. Net-negative on LOC, which
  also pays down some of the budget pressure. Skip this phase if you want to keep Streamlit as a
  fallback; say so and it drops.

## QA test plan

### Acceptance criteria (exit = all checked, per phase where noted)

- [ ] **AC-1 No-bypass.** Every console turn goes through `channel.run_turn` → `master.handle`; it
      is classified, governed, and persisted exactly like a typed turn (a `workflow_runs` row, a
      queue row). The SSE stream is observation only; it never drives orchestration.
- [ ] **AC-2 Roster fidelity (P01).** The panel's initial roster equals `WorkflowPlan.agents` for
      the turn; every agent's final state matches its `agent_runs` outcome; a peer-replaced agent
      shows retire + join. Verified against the trace, not the stream alone.
- [ ] **AC-3 Activity order (P02).** The streamed step sequence matches the documented event order
      for the chosen execution mode; the rendered answer equals `Response.text`.
- [ ] **AC-4 Stream isolation.** With the evolution/authoring daemons running, their events do not
      leak into a turn's console stream (single-flight name-filter in v1, or `stream_id` in v2).
- [ ] **AC-5 Approval in-UI (P03).** A gated turn shows Approve/Decline; Approve resolves via
      `master.resume_approval` and renders the real answer; Decline declines; `/pending` lists open
      approvals. Reuses Phase-03 seams, no re-implemented resume.
- [ ] **AC-6 Sources (P03).** The Sources panel shows the turn's `semantic_messages` + vault links
      and nothing it did not actually recall.
- [ ] **AC-7 SDK isolation.** FastAPI/uvicorn are imported only under `web/console/`; `service`/core
      import without the `[console]` extra; the suite is green with the extra absent (the SSE bridge
      tests drive the forwarder directly, no real HTTP).
- [ ] **AC-8 Additive.** REPL / one-shot / web / MCP / Telegram and all orchestration are
      byte-unchanged through Phase 03; no new core dependency. (Phase 04 changes `llm.py` behind a
      flag; Phase 05 removes Streamlit.)
- [ ] **AC-9 Ops surfaces.** `ubongo console` starts the server; missing extra → friendly install
      hint, rc 1, no traceback; ctl `start|stop|status console` works; systemd unit + start script
      present.
- [ ] **AC-10 Token-stream opt-in (P04).** With the flag off, `llm.complete()` is byte-identical;
      with it on, the composer streams and token accounting + the single-retry contract still hold.
- [ ] **AC-11 Full suite green** with `tests/test_console_service.py` + `tests/test_stream_bridge.py`;
      ADR-0021 + CHANGELOG `## v0.6.NN` per phase (version CI-derived).

### Regression plan (layered)

1. **Unit (the test surface is the bridge + the FastAPI handlers, HTTP not required).** The
   event-forwarder: given a synthetic sequence of bus events, the queue yields the right SSE frames
   in order; daemon-named events are filtered (AC-4). The roster reducer: `after_plan` +
   `agent_started/completed/failed` produce the expected panel states including peer replacement
   (AC-2). The turn handler runs `channel.run_turn` (mocked) in a task and closes the stream on
   `after_send`.
2. **Approval + sources.** A gated `Response` surfaces an Approve action that calls
   `resume_approval`; the sources panel reads a stub `RecallContext`. Both assert against the
   existing seams, not new ones.
3. **Deterministic surface.** `ubongo console` without the extra → hint + rc 1; help/usage.
4. **Live (manual, on the box).** Open the console, send a turn, watch the roster light up and the
   steps stream; run a debate-mode turn to see rounds; trigger a destructive prompt and approve it
   from the panel; confirm the Sources panel matches what `/recall` returns for the same query.
5. **Cumulative playbook** stays the contract; a new section **W** is this line's acceptance
   surface (W.1 stream a typed turn, W.2 roster matches the plan, W.3 peer-replacement renders,
   W.4 approval in-UI, W.5 sources match recall, W.6 daemon events do not leak, W.7 full pytest).

### Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Daemon/concurrent events leak into a turn's stream | wrong roster/steps shown | single-flight + event-name filter in v1; `stream_id` contextvar correlation in v2 (AC-4) |
| Background-task turn swallows an exception silently | console hangs "working" forever | the task wraps `channel.run_turn` in try/except, emits a terminal `error` SSE frame, always closes the stream (mirrors the daemon crash-swallow) |
| SSE connection drops mid-turn | user loses the live view but the turn still runs | the turn is unaffected (it is observation-only); on reconnect the console replays from the trace rows, which are the source of truth |
| FastAPI/uvicorn import leaks into core | suite needs the extra | imported only under `web/console/`; AC-7 asserts core imports clean |
| Token-streaming changes `llm.py` for everyone | a regression in the one model chokepoint | Phase 04 is flag-gated and optional; off = byte-identical; AC-10 locks the non-streaming contract |
| Two web UIs drift | maintenance burden, confusion | Phase 05 retires Streamlit; until then the console is additive and Streamlit is untouched |
| LOC budget (already ~7% over) grows | self-imposed budget breached further | Phase 00-03 add a channel (bounded); Phase 05 removes Streamlit to claw back; the frontend assets live outside `src/` |

## Why this fits the architecture

It adds no second path around any governed seam. Turns still go through `master.handle` via
`channel.run_turn`; durable writes still go through the Memory Agent; approval still resolves the
one `pending_approvals` record; the console only *observes* (the event bus and the trace) and
*resumes* (the existing approval seam). The single new primitive, per-turn event streaming, is a
handler on the named events, which is exactly how the project is meant to grow.
