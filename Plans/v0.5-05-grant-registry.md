# v0.5 Phase 05 — The grant registry

Phase 05 of [Plans/v0.5-trust-protocol.md](v0.5-trust-protocol.md) — the heart of the plan and
its one genuinely new subsystem. Branch `v0.5/05-grant-registry` → ships as **v0.5.5** (CI
derives the version from the branch name; do not hand-edit VERSION). Built **channel-free**:
the registry runs on the Phase 03 approval seam via the existing REPL/web/CLI surfaces, so it
needs no messaging channel. (The Telegram channel is a separate later phase; Signal/WhatsApp
after that.) The version sequence will read `0.5.3 → 0.5.5`; `0.5.4` is reserved for the
channel phase and is filled when that ships.

## The problem

Phase 03 made governance approvals persistent and resumable, but they are still **ask-every-
time**: every connector turn that touches a high-risk server re-prompts, even when you already
said yes to that capability a hundred times. That is the fatigue the trust-protocol plan set
out to kill. ADR-0016 deferred per-call gating "until real tool names exist"; this phase
delivers it at the granularity we *can* know before execution — the server.

## The change

A **grant** is persistent consent for a capability class. The first connector turn touching a
class with no active grant asks once (through the Phase 03 seam); approving writes the grant;
every later turn in that class auto-proceeds with a one-line narration; revoking re-arms the
ask. Persistent, restart-surviving, human-granted.

1. **`grants` table** (additive, `CREATE IF NOT EXISTS`, no migration): `id`,
   `capability_class` (e.g. `connector:compendium`), `consequence_class`
   (`reversible|irreversible`), `scope` (agent name or `*`), `purpose`, `status`
   (`active|revoked`), `created_at`, `revoked_at`. CRUD in a new `memory/grant_state.py`
   (the phase-02 state-module pattern), written by the orchestrator — same governance carve-out
   as `governance_decisions`, **not** the Memory Agent.
2. **`governance/grants.py`** + a new rule in `governance/decision.py`: for a connector turn,
   derive the capability class(es) from the enabled servers (`mcp.client.servers()`); if every
   class has an active grant → `AUTO` (with a narration note); if any lacks one →
   `REQUIRE_APPROVAL` reason `grant_first_encounter:<class>`. The rule sits **after** the
   destructive (rule 1) and irreversible-high (rule 2) safety rules — a destructive connector
   turn still gates on safety regardless of grants. Non-connector turns never touch the registry.
3. **Approval → grant link.** When an approved connector turn proceeds (the `approved=True`
   bypass) and lacks a grant for its class, an active grant is persisted, so the *next* turn
   auto-proceeds. Approving a first-encounter connector turn = granting its capability class.
4. **Management surface.** REPL `/grants` (list active) + `/grants revoke <id>`; CLI
   `ubongo grants` + `ubongo grants revoke <id>`. DB-backed, so revocation survives restart.
5. **Chief-of-Staff-as-config (minimal).** The `scope` column exists; the enforced invariant is
   "a capability requires a grant, period — an agent cannot exercise one the registry lacks."
   Full chartered-role routing config is documented in the ADR, **not** built as a system here.
6. **Paired cut (Amendment 2).** Delete the `retry:repair` evolvable target and its
   structural-proxy fitness — the weakest evolution signal (ADR-0007) — as the budget offset
   for the new subsystem.
7. **ADR-0019** records the decision: grant-checked-at-decision for connector turns (extends
   ADR-0004 + ADR-0016), server-granularity classes, the per-tool-name deferral, and the cut.

**Deferred (consistent with ADR-0016):** per-tool-name allowlists within a server. Grants are
server-granular until a real integration (Calendar/Gmail/Compendium) gives concrete tool names.

## QA test plan (prepared with /qa-test-planner)

### Acceptance criteria (exit = all checked)

- [ ] **AC-1 Ask-once (the exit criterion).** A connector turn touching a class with no grant →
      `require_approval` (`grant_first_encounter:<class>`); approving writes an active grant and
      delivers; the **next** connector turn in that class → `auto`, no prompt. Headlessly
      testable.
- [ ] **AC-2 Revoke re-arms.** Revoking the grant flips it to `revoked`; the next connector turn
      in that class gates again. Survives a fresh process (DB-backed).
- [ ] **AC-3 Safety still wins.** A destructive connector turn gates on rule 1 even with an
      active grant — grants never override the safety rules.
- [ ] **AC-4 Non-connector turns untouched.** auto / reject / ask_clarification, the four
      personas, all six execution modes: no grant lookup, no grant rows written, decisions
      identical to pre-phase. (Behaviour-neutral.)
- [ ] **AC-5 Approval→grant link.** The grant is written when the approved connector turn
      proceeds, keyed to the capability class derived from the enabled servers; declining writes
      no grant.
- [ ] **AC-6 Management surface.** `/grants` and `ubongo grants` list active grants;
      `/grants revoke <id>` and `ubongo grants revoke <id>` revoke; an unknown id is a clean
      no-op message.
- [ ] **AC-7 Narration.** A granted connector turn surfaces a one-line "proceeding under grant
      #N (<class>)" note (in the trace / decision reason), so auto-proceed is never silent.
- [ ] **AC-8 retry:repair cut.** The `retry:repair` target is gone from `/optimize`'s list and
      from the evolvable-targets registry; its fitness proxy is deleted; no test references it;
      `src/` LOC is net-down or flat versus the registry's additions.
- [ ] **AC-9 Schema + single-writer.** `grants` is additive (`CREATE IF NOT EXISTS`); the
      strict-mode single-writer test passes unmodified.
- [ ] **AC-10 Full suite green**, new tests for the registry + governance rule + management +
      the cut, ADR-0019 + CHANGELOG `## v0.5.5` shipped (version CI-derived).

### Regression plan (layered)

1. **Unit.** `grant_state` CRUD (append/get-active/list/revoke; idempotent revoke);
   `governance/grants.py` capability-class derivation from enabled servers; the new decision
   rule in isolation (grant present → auto; absent → require_approval; destructive → still
   gates).
2. **Master integration.** A connector turn with no grant gates; `resume_approval("y")` writes
   the grant + delivers; the next connector turn autos; revoke → re-gates; decline → no grant.
   Non-connector turns assert no grant rows (behaviour-neutral).
3. **Surface.** REPL `/grants` list/revoke handlers (pure); CLI `ubongo grants` subcommands.
4. **The cut.** `/optimize` no longer lists `retry:repair`; the evolvable-targets set excludes
   it; the deleted fitness path has no dangling refs (grep-clean).
5. **Deterministic + live mini-sweep.** `/grants` empty + populated; a live connector turn
   (loop-back MCP server) gated → granted → second turn autos; `/decisions` shows the grant
   reason.
6. **Cumulative playbook** sections 0–21/A/P/M/C/L/S/T stay the regression contract; new
   section **U** is this phase's acceptance surface.

### Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Capability class differs decision-time vs the tool actually called | a grant covers more than intended | class = server (the unit we know pre-execution); ADR-0019 states server-granularity explicitly; per-tool allowlists deferred to a real integration |
| Grant silently overrides a safety gate | a destructive call auto-proceeds | the grant rule is placed AFTER rules 1–2; AC-3 locks it with a test |
| The approval→grant write fires on decline or on a non-connector turn | spurious grants | grant write is gated on (connector turn AND approved AND no existing grant); AC-4/AC-5 assert both negatives |
| Removing `retry:repair` breaks evolution config/tests | suite red | grep every reference before deleting; the cut is its own commit with the suite green after |
| Circular import (governance ↔ grant_state ↔ master) | import failure | `grant_state` is pure store CRUD; `governance/grants.py` reads it; master writes on approval — one direction |
