# v0.5 Phase 04 — The Telegram channel

Phase 04 of [Plans/v0.5-trust-protocol.md](v0.5-trust-protocol.md). Branch
`v0.5/04-telegram` → fills the reserved **v0.5.4** slot (CI derives the version from the branch
name; do not hand-edit VERSION). The first *cloud-relayed* channel, and the delivery path the
rest of the trust protocol was waiting on: it's how approve-later (Phase 03) and grant
first-encounter asks (Phase 05) reach you remotely, and how standing jobs (Phase 06) will
"park and raise."

## The shape

A channel is a presentation layer over the one turn envelope (`channel.run_turn` →
`master.handle` → queue flush). Telegram is an additive adapter over it — the closest analog
is the MCP channel, and this mirrors its structure exactly:

- **`telegram/service.py`** — the channel-free, **network-free, unit-testable** core. Auth
  (`is_allowed(user_id)` against `telegram.allowed_user_ids`) and
  `handle_message(text, user_id) -> reply` that either (a) refuses an unauthorized user,
  (b) handles `/approve|/decline|/pending|/grants` via the existing seams
  (`master.resume_approval`, `grant_state`), or (c) runs `channel.run_turn(text, persona,
  auto_mode=True)` and, when `response.approval is not None`, appends the gated prompt naming
  the `decision_id`. No bypass: every turn goes through `master.handle`.
- **`telegram/bot.py`** — the **only** module touching the Telegram Bot API: a thin `httpx`
  long-poll loop (`getUpdates` / `sendMessage`), lazy import, token from `TELEGRAM_BOT_TOKEN`
  in `.env`. `ubongo telegram` entrypoint in `__main__`.

## What changes

1. The `telegram/` package (service + bot) and the `ubongo telegram` entrypoint.
2. **Auth returns.** `telegram.allowed_user_ids` in `settings.yaml` — a list of Telegram user
   ids; **empty = deny all (fail-closed)**. The first channel since v0.1 where auth is real
   (the v0.1 plan reserved `allowed_user_ids` for exactly this).
3. **A `before_send` policy handler seam**, wired on the named event. Minimal here
   (default-allow); the full quiet-hours / holds / catch-up policy engine is a later phase.
   Telegram delivery is policy-gated additively, not by editing the core.
4. **Optional extra `[telegram]`** (httpx), lazily imported only in `bot.py`, kept out of core
   — the suite runs without it, exactly like `streamlit` / `mcp`. ctl + systemd unit +
   `start-ubongo-telegram.sh` + installer support, mirroring MCP. `api.telegram.org` added to
   `deploy/envelope/egress.hosts` (ADR-0017).
5. **ADR-0020** records the trust-posture change: the first cloud-relayed channel, auth
   returns, the Phase-01 egress envelope is what makes it acceptable, approve-later rides the
   Phase-03 seam, no orchestration bypass.
6. The "Telegram" docs (CLAUDE.md, the v0.5 plan) are now correct again (the channel
   preference reverted to Telegram); no retarget needed.

## QA test plan (prepared with /qa-test-planner)

### Acceptance criteria (exit = all checked)

- [ ] **AC-1 No-bypass.** Every authorized Telegram turn goes through `channel.run_turn` →
      `master.handle`; it is classified, governed, and persisted exactly like a typed turn
      (a `workflow_runs` row, a queue row).
- [ ] **AC-2 Auth fail-closed.** An unconfigured/empty `allowed_user_ids` denies everyone; an
      unlisted user id gets a refusal and **no turn runs** (no `workflow_runs` row); a listed
      id is served.
- [ ] **AC-3 Approve-later over Telegram (the point).** A gated turn replies with the gated
      message **and the decision_id**; `/approve <id>` resolves it via `master.resume_approval`
      and delivers the real answer; `/decline <id>` declines. `/pending` lists open approvals;
      `/grants` lists active grants. (Reuses Phase 03/05 — no re-implemented resume.)
- [ ] **AC-4 SDK isolation.** `httpx`/the Bot API is imported only in `bot.py`; `service.py`
      and the whole core import without the `[telegram]` extra; the test suite is green with
      the extra absent (the bot loop's tests mock the transport).
- [ ] **AC-5 before_send seam.** A registered `before_send` handler can suppress/hold a
      Telegram delivery without touching the core; default is allow (behaviour-neutral).
- [ ] **AC-6 Secrets + config.** The token is read only from `TELEGRAM_BOT_TOKEN` in `.env`
      (never config, never logged); `allowed_user_ids` lives in `settings.yaml`.
- [ ] **AC-7 Additive.** REPL / one-shot / web / MCP and all orchestration are byte-unchanged;
      no new core dependency (the extra is optional).
- [ ] **AC-8 Ops surfaces.** `ubongo telegram` starts the long-poll loop; missing extra →
      friendly install hint, rc 1, no traceback; ctl `start|stop|status telegram` works;
      systemd unit + start script present; `api.telegram.org` in the egress allowlist.
- [ ] **AC-9 Full suite green** with new `tests/test_telegram_service.py`; ADR-0020 + CHANGELOG
      `## v0.5.4` shipped (version CI-derived).

### Regression plan (layered)

1. **Unit (the test surface is `service.py`).** `is_allowed` (empty/unlisted/listed); the
   command router (`/approve`, `/decline`, `/pending`, `/grants`, unknown); `handle_message`
   for a normal turn (mock `channel.run_turn` → asserts the reply) and a gated turn (asserts
   the decision_id is surfaced); the refusal path writes no turn.
2. **Bot loop (transport mocked).** `bot.py`'s update→reply pump with `httpx` patched: a parsed
   `getUpdates` payload drives `service.handle_message` and a `sendMessage` call; the missing
   extra produces the friendly hint.
3. **Deterministic surface.** `ubongo telegram` without the extra → hint + rc 1; help/usage.
4. **Live (manual / needs a token).** Real bot: an authorized chat round-trips a turn; an
   unlisted user is refused; a destructive prompt gates and `/approve <id>` delivers; a
   connector first-encounter asks and is granted from the phone. (Pi-only, like the MCP HTTP
   and envelope checks.)
5. **Cumulative playbook** sections 0–21/A/P/M/C/L/S/T/U stay the contract; new section **V**
   is this phase's acceptance surface.

### Smoke playbook section (lands as section V)

V.1 `ubongo telegram` without the extra → friendly hint, rc 1 · V.2 unit: unlisted user
refused, no turn · V.3 unit: authorized normal turn round-trips through `master.handle` ·
V.4 unit: gated turn surfaces the decision_id; `/approve <id>` delivers · V.5 `/pending` +

### Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Auth defaults open | anyone messaging the bot drives the agent | empty `allowed_user_ids` = deny all; AC-2 locks fail-closed with a test |
| Token leaks into logs/config | credential exposure | token only from `.env`, never logged; AC-6; the bot logs user ids, never the token |
| SDK import leaks into core | suite needs the extra | httpx imported only inside `bot.py` functions; AC-4 asserts core imports clean |
| Cloud relay weakens the local-first posture | the trust-frame shifts | ADR-0020 names it; the Phase-01 egress envelope + `allowed_user_ids` are the controls; documented, not hidden |
| Long-poll loop wedges on a transient API error | bot goes silent | per-iteration try/except with backoff in the loop (mirrors the daemon crash-swallow); the loop logs and continues |
| A gated turn's approval is approved by a different chat | wrong principal approves | resume is by decision_id; only allowed users reach the command router at all (AC-2) — a second auth layer on /approve is a follow-up if multiple users are ever allowed |
