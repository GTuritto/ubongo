# v0.5 Phase 03 — The typed, resumable approval seam

Phase 03 of [Plans/v0.5-trust-protocol.md](v0.5-trust-protocol.md); the architecture review's
candidate 4. Branch `v0.5/03-approval-seam` → ships as **v0.5.3** (CI derives the version from
the branch name on merge; do not hand-edit VERSION). This is the first v0.5 phase that changes
runtime behavior rather than refactoring/infra, and it is the load-bearing prerequisite for
approve-later over Telegram (Phase 04) and standing jobs (Phase 06).

## The problem

When the decision matrix returns `require_approval`, `master` attaches `Response.approval` as a
plain dict `{decision_id, summary, why}` and each channel hand-rolls the resume:
- **REPL** prompts y/n/why, then `trace.update_governance_decision` + re-issue `handle(approved=True)`.
- **Web** stashes the whole turn (`message`/`persona`/`auto_mode`) in `st.session_state` and
  does the same on a button.
- **One-shot** prints the gated message and exits 1 — no resume.
- **MCP** returns `gated=True` — no resume.

Resume only works while the *requesting channel still holds the turn in memory*. Nothing is
persisted, so a turn gated in one channel cannot be approved in another, and a process restart
loses it. That is exactly the wall Telegram and proactive jobs would hit.

## The change

The pending-approval **record** becomes the single source of truth for resume.

1. **`pending_approvals` table** (additive, `CREATE IF NOT EXISTS`, no migration): `decision_id`
   PK (→ `governance_decisions.id`), `message`, `persona`, `auto_mode`, `summary`, `why`,
   `status` (`pending|approved|declined`), `created_at`, `resolved_at`. CRUD lives in
   `memory/trace.py` next to `governance_decisions` (they are keyed together and resolved
   together): `append_pending_approval`, `get_pending_approval`, `open_pending_approvals`,
   `resolve_pending_approval`.
2. **`governance/approval.py`**: keep `ApprovalRequest` + `build_request` + `explain`; add a
   frozen `PendingApproval` dataclass and thin typed read wrappers. No `master` import (avoids
   the cycle).
3. **`Response.approval` becomes typed `ApprovalRequest | None`** (was `dict`). On every
   `require_approval` turn `master` persists a pending record (message, persona, auto_mode,
   summary, why) in addition to attaching the request.
4. **`master.resume_approval(decision_id, choice, *, profile_cpu=False) -> Response | None`** —
   the one resume function: it reads the persisted record, resolves it + writes
   `governance_decisions.approval_response`, and on `"y"` re-issues
   `handle(message, persona, auto_mode, approved=True)` **from the record** (not channel
   memory), returning the delivered Response; on `"n"` returns `None`. Idempotent: a record that
   is already resolved (or absent) returns `None` without re-running.
5. **Channels become thin adapters** over the seam:
   - REPL: in-session gated turn → `master.resume_approval`; new `/pending` (list) and
     `/pending approve|decline <id>` for turns raised elsewhere.
   - Web: holds only the `decision_id`; reads message/persona from the record; resumes via
     `master.resume_approval`. No turn in session state.
   - One-shot CLI: gated turn still exits 1, but the record now persists; add `ubongo pending`
     and `ubongo approve|decline <id>` subcommands.
   - MCP: `gated=True` surfaces the `decision_id`; the persisted record lets a human channel
     approve later (still never approvable over MCP itself — ADR-0015).

**Invariant established:** the `pending_approvals` record is the single source of truth for
resume; no channel re-implements the resume contract or holds the gated turn in session state.
Recorded in **ADR-0018**.

## QA test plan (prepared with /qa-test-planner)

### Acceptance criteria (exit = all checked)

- [ ] **AC-1 Cross-channel resume (the exit criterion).** A turn gated via one-shot (or MCP)
      persists a `pending_approvals` row; `master.resume_approval(id, "y")` — invoked with no
      reference to the original channel — delivers the real answer and marks the record
      `approved`. Headlessly testable, no Streamlit/REPL process required.
- [ ] **AC-2 Typed seam.** `Response.approval` is an `ApprovalRequest` (frozen dataclass), not a
      dict; the four channels read `.summary`/`.why`/`.decision_id`, and MCP serializes it.
- [ ] **AC-3 Record is source of truth.** `resume_approval` re-issues from the persisted
      message/persona/auto_mode; the web channel no longer stores the turn in `session_state`
      (only the `decision_id`).
- [ ] **AC-4 Resolve writes both rows.** Approving/declining sets `pending_approvals.status` +
      `resolved_at` *and* `governance_decisions.approval_response`, in one call.
- [ ] **AC-5 Idempotent / safe.** `resume_approval` on an unknown, already-approved, or
      already-declined `decision_id` returns `None` and re-runs nothing (no duplicate turn, no
      second answer).
- [ ] **AC-6 Behaviour-neutral elsewhere.** `auto`, `reject`, and `ask_clarification` turns are
      unchanged — no pending record written, `Response.approval is None`.
- [ ] **AC-7 In-session parity.** The REPL y/n/why prompt and the web Approve/Deny buttons still
      work end-to-end (now through `resume_approval`); `why` still re-prompts without resolving.
- [ ] **AC-8 Listable surface.** `/pending` lists open approvals (id, summary, age);
      `/pending approve|decline <id>` resolves them; `ubongo pending` / `ubongo approve <id>`
      do the same from the CLI.
- [ ] **AC-9 Schema + single-writer intact.** `pending_approvals` is additive (`CREATE IF NOT
      EXISTS`); the strict-mode single-writer test passes unmodified (governance state follows
      the `governance_decisions` carve-out, not the Memory Agent).
- [ ] **AC-10 Full suite green**, with new tests for the seam + resume + cross-channel, and the
      CHANGELOG `## v0.5.3` entry shipped (version itself is CI-derived).

### Regression plan (layered)

1. **Unit (the test surface is the seam).** `governance/approval.py` + the `trace.py` CRUD:
   build_request/explain unchanged; record round-trips; `open_pending_approvals` ordering;
   resolve transitions (pending→approved, pending→declined) and idempotency.
2. **Master integration.** `require_approval` writes a pending row + typed `approval`;
   `resume_approval("y")` delivers and flips both rows; `resume_approval("n")` declines, no
   delivery; unknown/double-resolve → `None`. The `auto`/`reject`/`ask_clarification` paths
   write no pending row (behaviour-neutral assertions).
3. **Channel adapters.** REPL `_prompt_approval` + `/pending` handlers (pure, tested directly);
   web gate reads the record (AppTest or a unit over the render fn); the one-shot CLI
   subcommands parse + dispatch.
4. **Deterministic surface pass.** `/pending` (empty + populated), `/policy`, `/decisions`,
   `/trace` render unchanged; help banner lists `/pending`.
5. **Live mini-sweep (P0).** Gate a destructive turn in one-shot (exit 1, record persisted) →
   `ubongo approve <id>` delivers the real answer → `/decisions` shows `approval_response`.
6. **Cumulative playbook** sections 0–21/A/P/M/C/L/S stay the regression contract; new section
   **T** (below) is this phase's acceptance surface.

### Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Circular import (`governance/approval` ↔ `master`) | import failure | `resume_approval` lives in `master`; `governance/approval` owns only types + read wrappers, never imports master |
| Typed `approval` breaks a channel that indexed the dict | runtime AttributeError in a rarely-hit path | grep every `approval[` / `.approval` access across the 4 channels + tests before merge; the deterministic surface pass exercises each |
| A resumed turn re-gates (infinite gate) | user can never approve | `approved=True` bypass is unchanged; AC-5 idempotency test covers re-entry |
| Pending record leaks the message text into a new table | privacy surface | same store, same single-writer posture as `messages`; documented in ADR-0018; no new external exposure |
| `auto_mode` not preserved on resume | resumed turn routes differently than intended | record stores `auto_mode`; resume passes it through; asserted in AC-3 |
